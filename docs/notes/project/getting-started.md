# Getting Started With beyondwin

This tutorial gets the local `beyondwin` site running, checks the content model,
and shows where the main publishing surfaces live.

## What You'll Need

- Node.js and npm.
- A local checkout of this repository.
- Dependencies installed with `npm install` if `node_modules/` is missing.

The project is an Astro and MDX static site. Astro builds routes from files in
`src/pages/`, and content collections from MDX files in `src/content/`.

## Step 1: Install Dependencies

From the repo root:

```bash
npm install
```

This installs Astro, MDX support, TypeScript, Vitest, `gray-matter`, and `yaml`.

## Step 2: Run The Full Validation Gate

```bash
npm run validate
```

This is the preferred pre-commit check. It runs content validation, article
quality checks, memory projection validation, tests, Astro type checking, and the
static build.

Expected success shape:

```text
Content validation passed.
Article quality validation passed.
Memory projection valid. thoughts=<n> topics=<n> edges=<n> sources=<n> excluded={...}
```

Astro and Vitest print their own success output after those lines.

## Step 3: Start The Local Site

```bash
npm run dev
```

Open the URL printed by Astro, usually:

```text
http://localhost:4321/
```

The home page should show the article desk, archive counts, route lanes, tag
rack, and latest entries.

## Step 4: Visit The Main Routes

Check these paths while the dev server is running:

- `/articles/` for development articles and technical essays.
- `/analysis/` for source-grounded queue output.
- `/reviews/` for book, tool, course, article, and other reviews.
- `/ideas/` for early idea notes.
- `/travel/` for travel notes.
- `/memory/` for the public projection of reviewed memory thoughts.
- `/tags/` for the tag index.

Collection detail routes use the file slug from `src/content/<collection>/`.
For example, `src/content/articles/example-article.mdx` becomes
`/articles/example-article/`.

## Step 5: Trace One Piece Of Content

Open [src/content/articles/example-article.mdx](../../../src/content/articles/example-article.mdx).

Its frontmatter is checked by [src/content.config.ts](../../../src/content.config.ts)
and [scripts/validate-content.mjs](../../../scripts/validate-content.mjs).
Its detail page is generated by
[src/pages/articles/[slug].astro](../../../src/pages/articles/[slug].astro).
Its preview card is rendered by
[src/components/ContentCard.astro](../../../src/components/ContentCard.astro).

That is the normal content path:

```text
MDX file -> content collection schema -> content helper -> route -> layout/component
```

## What You Built

You now have the site running locally, the validation gate passing, and a concrete
map from one MDX file to the rendered route. Use
[How to Publish And Maintain Content](publishing-workflows.md) for day-to-day
content tasks, and [Project Architecture Reference](architecture-reference.md)
when you need exact schemas, scripts, and route contracts.
