# 섹션 2. 트랜잭션

---

## 4. 트랜잭션을 사용할 때 주의할 점이 있을까요?

### 예상 질문

> 트랜잭션을 사용할 때 어떤 점을 주의해야 하나요?

### 핵심 답변

트랜잭션은 데이터 정합성을 보장하기 위한 수단이지만, 무조건 많이 쓰거나 길게 유지하면 오히려 성능과 장애의 원인이 됩니다. 핵심은 **트랜잭션 범위를 작게 유지하는 것**입니다.

주의할 점은 크게 네 가지입니다.

1. **트랜잭션 안에서 외부 API 호출을 피해야 합니다.**
2. **트랜잭션 시간을 짧게 유지해야 합니다.**
3. **락 경합이 발생할 수 있는 쿼리를 조심해야 합니다.**
4. **읽기 작업과 쓰기 작업을 구분해야 합니다.**

예를 들어 주문 생성 트랜잭션 안에서 결제 PG API를 호출하면, PG 응답이 3초 지연되는 동안 DB 락과 커넥션을 계속 점유하게 됩니다. 트래픽이 몰리면 커넥션 풀이 고갈되고, 다른 요청까지 장애가 납니다.

### 실무 사례

나쁜 구조:

```text
트랜잭션 시작
→ 주문 INSERT
→ 재고 UPDATE
→ 외부 결제 API 호출
→ 결제 결과 저장
→ 트랜잭션 커밋
```

이 구조는 외부 API가 느려질 때 DB 커넥션과 락을 오래 붙잡습니다.

개선 구조:

```text
트랜잭션 1
→ 주문 PENDING 생성
→ 커밋

외부 결제 API 호출

트랜잭션 2
→ 결제 성공 시 PAID 변경
→ 재고 차감 또는 예약 확정
→ 커밋
```

### 면접 답변 예시

> 트랜잭션은 정합성을 위해 필요하지만, 범위를 최소화해야 합니다. 특히 외부 API 호출, 파일 업로드, 메시지 발행 같은 비DB 작업을 트랜잭션 내부에 넣으면 커넥션 점유 시간이 길어지고 락 경합이 커질 수 있습니다. 그래서 저는 트랜잭션 내부에는 DB 상태 변경만 두고, 외부 연동은 트랜잭션 전후로 분리하거나 Outbox 패턴을 사용합니다.

## 5. 트랜잭션 격리 수준의 종류를 설명해주세요

### 예상 질문

> 트랜잭션 격리 수준에는 어떤 것들이 있나요?

### 핵심 답변

트랜잭션 격리 수준은 동시에 여러 트랜잭션이 실행될 때, 한 트랜잭션이 다른 트랜잭션의 변경을 어디까지 볼 수 있는지를 정하는 기준입니다.

대표적으로 네 가지가 있습니다.

| 격리 수준 | 특징 | 발생 가능한 문제 |
|---|---|---|
| `READ UNCOMMITTED` | 커밋되지 않은 데이터도 읽음 | Dirty Read |
| `READ COMMITTED` | 커밋된 데이터만 읽음 | Non-Repeatable Read, Phantom Read |
| `REPEATABLE READ` | 같은 트랜잭션 안에서 같은 row는 동일하게 읽힘 | DB 구현에 따라 Phantom Read 가능 |
| `SERIALIZABLE` | 트랜잭션을 직렬 실행처럼 보장 | 성능 저하, 락 경합 증가 |

MySQL InnoDB는 기본적으로 `REPEATABLE READ`를 사용하고, PostgreSQL은 일반적으로 `READ COMMITTED`를 기본으로 사용합니다.

### 실무 사례

게시글 조회수 같은 단순 업데이트는 `READ COMMITTED`로도 충분할 수 있습니다. 반면 계좌 이체, 재고 차감, 쿠폰 발급처럼 정합성이 중요한 작업은 격리 수준만 믿지 말고 **명시적 락, 유니크 제약, 낙관적 락, 비관적 락**을 함께 고려해야 합니다.

### 면접 답변 예시

> 격리 수준은 동시성 상황에서 데이터 일관성과 성능 사이의 트레이드오프를 조절하는 옵션입니다. `READ COMMITTED`는 성능이 좋지만 같은 트랜잭션 안에서도 다시 읽으면 값이 달라질 수 있고, `REPEATABLE READ`는 같은 row에 대한 반복 읽기를 보장합니다. `SERIALIZABLE`은 가장 강하지만 락 경합과 성능 저하가 커서 실무에서는 꼭 필요한 경우에만 사용합니다.

## 6. Dirty Read, Non-Repeatable Read, Phantom Read를 설명해주세요

### 예상 질문

> 격리 수준에 따라 발생할 수 있는 문제를 설명해주세요.

### Dirty Read

다른 트랜잭션이 아직 커밋하지 않은 데이터를 읽는 문제입니다.

```text
T1: user.balance = 100 → 50으로 변경
T2: 50을 읽음
T1: 롤백
```

T2는 실제로 존재하지 않았던 값을 읽은 셈입니다.

### Non-Repeatable Read

같은 트랜잭션 안에서 같은 row를 두 번 읽었는데 값이 달라지는 문제입니다.

```text
T1: 회원 A의 등급을 읽음 → SILVER
T2: 회원 A 등급을 GOLD로 변경 후 커밋
T1: 다시 읽음 → GOLD
```

### Phantom Read

같은 조건으로 조회했는데, 두 번째 조회에서 없던 row가 생기거나 사라지는 문제입니다.

```text
T1: price > 10000 상품 조회 → 10개
T2: price > 10000 상품 추가 후 커밋
T1: 다시 조회 → 11개
```

### 실무 사례

정산 시스템에서 “이번 달 정산 대상 주문”을 조회한 뒤 처리하는 중간에 다른 트랜잭션이 정산 대상 주문을 추가하면 Phantom Read가 발생할 수 있습니다. 이 경우 단순 `SELECT`만으로는 안전하지 않고, 상태값 전이, 배치 기준 시점 고정, 락, 스냅샷 기준 등을 함께 설계해야 합니다.

### 면접 답변 예시

> Dirty Read는 커밋되지 않은 데이터를 읽는 문제이고, Non-Repeatable Read는 같은 row를 반복 조회했을 때 값이 달라지는 문제입니다. Phantom Read는 같은 조건으로 범위 조회를 반복했을 때 row 집합 자체가 달라지는 문제입니다. Dirty Read는 `READ COMMITTED` 이상에서 방지되고, 반복 읽기와 팬텀 문제는 더 높은 격리 수준이나 명시적 락, 범위 락을 고려해야 합니다.

## 7. 트랜잭션 격리 수준에 따라 나타나는 문제를 어떻게 해결하나요?

### 예상 질문

> Dirty Read, Non-Repeatable Read, Phantom Read는 어떻게 해결할 수 있나요?

### 핵심 답변

해결 방식은 문제 유형별로 다릅니다.

| 문제 | 해결 방향 |
|---|---|
| Dirty Read | `READ COMMITTED` 이상 사용 |
| Non-Repeatable Read | `REPEATABLE READ` 이상 사용 또는 명시적 락 |
| Phantom Read | `SERIALIZABLE`, next-key lock, predicate lock, 명시적 범위 락 |
| Lost Update | 격리 수준만으로 부족할 수 있어 낙관적/비관적 락 필요 |

### 실무 사례

선착순 쿠폰 100개 발급에서 단순히 다음처럼 하면 위험합니다.

```sql
SELECT count(*)
FROM coupon_issued
WHERE coupon_id = 1;

INSERT INTO coupon_issued (...);
```

동시에 여러 요청이 들어오면 모두 `count = 99`로 보고 100개를 초과 발급할 수 있습니다.

개선 방식은 다음과 같습니다.

```sql
UPDATE coupon
SET issued_count = issued_count + 1
WHERE id = 1
  AND issued_count < max_count;
```

그리고 affected row가 1이면 성공, 0이면 마감으로 처리합니다.

### 면접 답변 예시

> 격리 수준을 올리는 것도 방법이지만, 실무에서는 성능 비용이 크기 때문에 문제의 성격에 맞게 해결합니다. Dirty Read는 `READ COMMITTED` 이상으로 해결하고, Lost Update나 선착순 발급 같은 문제는 낙관적 락, 비관적 락, 조건부 `UPDATE`, 유니크 제약을 함께 사용합니다.

## 8. MVCC가 무엇이고, 왜 사용하나요?

### 예상 질문

> MVCC가 무엇인가요?

### 핵심 답변

MVCC는 **Multi-Version Concurrency Control**, 즉 다중 버전 동시성 제어입니다. 하나의 row에 대해 여러 버전을 관리해서, 읽기 작업이 쓰기 작업을 막지 않고, 쓰기 작업도 읽기 작업을 최대한 막지 않도록 하는 방식입니다.

락 기반 동시성 제어만 사용하면 읽기와 쓰기가 서로 자주 막힙니다. MVCC를 사용하면 읽기 트랜잭션은 특정 시점의 스냅샷을 읽고, 쓰기 트랜잭션은 새로운 버전을 만듭니다.

### 실무 사례

상품 상세 페이지는 초당 수천 번 조회되고, 관리자는 가끔 상품 가격을 수정한다고 가정합니다.

MVCC가 없다면 가격 수정 중 조회가 막히거나, 조회 중 수정이 막힐 수 있습니다. MVCC를 사용하면 사용자는 기존 스냅샷을 읽고, 관리자의 수정은 새로운 버전으로 반영됩니다.

### 주의점

MVCC도 비용이 있습니다.

- 오래 열린 트랜잭션이 있으면 이전 버전을 정리하지 못할 수 있습니다.
- PostgreSQL에서는 vacuum 부담이 커질 수 있습니다.
- MySQL InnoDB에서는 undo log 보관 부담이 커질 수 있습니다.
- 장기 트랜잭션은 스토리지와 성능 문제를 유발할 수 있습니다.

### 면접 답변 예시

> MVCC는 row의 여러 버전을 관리해서 읽기와 쓰기의 충돌을 줄이는 동시성 제어 방식입니다. 읽기는 특정 시점의 스냅샷을 보고, 쓰기는 새로운 버전을 생성합니다. 그래서 락 경합을 줄이고 높은 동시성을 얻을 수 있습니다. 다만 오래 열린 트랜잭션이 있으면 이전 버전을 정리하지 못해 undo 영역이나 vacuum 부담이 커질 수 있습니다.

## 9. 왜 InnoDB는 REPEATABLE READ만으로도 Phantom Read를 해결할 수 있나요?

### 예상 질문

> 일반적으로 Phantom Read는 SERIALIZABLE에서 막는다고 하는데, MySQL InnoDB는 왜 REPEATABLE READ에서도 막을 수 있나요?

### 핵심 답변

MySQL InnoDB는 `REPEATABLE READ`에서 **next-key lock**을 사용합니다. next-key lock은 **record lock + gap lock**의 조합입니다.

즉, 존재하는 인덱스 레코드만 잠그는 것이 아니라, 인덱스 레코드 사이의 gap까지 잠급니다. 그래서 다른 트랜잭션이 해당 범위에 새로운 row를 insert하는 것을 막을 수 있습니다.

참고: <https://dev.mysql.com/doc/refman/8.2/en/innodb-locking.html>

### 실무 사례

```sql
SELECT *
FROM reservation
WHERE room_id = 10
  AND start_time < '2026-05-01 12:00'
  AND end_time > '2026-05-01 10:00'
FOR UPDATE;
```

이 쿼리로 특정 시간대 예약 충돌 여부를 확인한다고 가정합니다. 인덱스가 적절히 잡혀 있으면 InnoDB는 해당 범위를 잠가서 다른 트랜잭션이 같은 시간대 예약을 끼워 넣는 것을 막을 수 있습니다.

단, 인덱스가 없거나 조건이 인덱스를 제대로 타지 않으면 락 범위가 과도하게 커질 수 있습니다.

### 면접 답변 예시

> InnoDB는 `REPEATABLE READ`에서 단순 record lock뿐 아니라 gap lock을 포함한 next-key lock을 사용합니다. 그래서 범위 조회 조건에 해당하는 인덱스 구간과 그 사이 gap까지 잠글 수 있고, 다른 트랜잭션이 그 범위에 새로운 row를 삽입하는 것을 막아 Phantom Read를 방지할 수 있습니다. 다만 인덱스 설계가 잘못되면 락 범위가 넓어질 수 있습니다.

- Spring `@Transactional`은 언제 적용되지 않나요?

### 실무 사례: DB 커밋과 메시지 발행의 불일치

주문 생성 후 Kafka 메시지를 발행하는 상황에서 다음 문제가 생길 수 있습니다.

```text
DB 커밋 성공
→ Kafka 발행 실패
→ 주문은 생성됐지만 후속 배송/알림 이벤트 누락
```

```text

→ DB 커밋 실패
→ 존재하지 않는 주문 이벤트 발행
```

이런 문제를 막기 위해 Outbox 테이블에 이벤트를 함께 저장하고, 별도 워커가 안정적으로 메시지를 발행하는 구조를 사용할 수 있습니다.

### 면접 답변 예시

> 트랜잭션 안에서 외부 메시지를 바로 발행하면 DB 커밋과 메시지 발행 사이의 원자성을 보장하기 어렵습니다. 그래서 주문 상태 변경과 이벤트 저장을 같은 트랜잭션으로 처리하고, 이후 별도 프로세스가 Outbox 이벤트를 읽어 메시지 브로커로 발행하는 구조를 사용할 수 있습니다.

## 11. DB Lock이란 무엇이고, 어떤 종류가 있나요?

### 예상 질문

> DB Lock이 무엇이고 어떤 종류가 있나요?

### 핵심 답변

DB Lock은 동시에 여러 트랜잭션이 같은 데이터에 접근할 때 데이터 정합성을 지키기 위해 접근을 제어하는 장치입니다.

대표적으로 다음과 같습니다.

| 락 종류 | 설명 |
|---|---|
| Shared Lock | 읽기 락. 다른 읽기는 허용하지만 쓰기는 제한 |
| Exclusive Lock | 쓰기 락. 다른 읽기/쓰기 제한 |
| Row Lock | 특정 row 단위 락 |
| Table Lock | 테이블 전체 락 |
| Gap Lock | 인덱스 레코드 사이 공간 락 |
| Next-Key Lock | Record Lock + Gap Lock |
| Intention Lock | 테이블 수준에서 row lock 의도를 표시 |

### 실무 사례

계좌 이체에서 출금 계좌와 입금 계좌를 동시에 수정해야 한다면, 두 계좌 row에 대해 락을 잡아야 합니다.

```sql
SELECT *
FROM account
WHERE id IN (1, 2)
FOR UPDATE;
```

이때 항상 같은 순서로 락을 잡지 않으면 데드락이 발생할 수 있습니다.

### 면접 답변 예시

> DB Lock은 동시 접근 상황에서 데이터 정합성을 보장하기 위한 제어 장치입니다. 읽기 작업에는 공유 락, 쓰기 작업에는 배타 락이 사용될 수 있고, InnoDB에서는 row lock, gap lock, next-key lock 같은 락도 사용됩니다. 실무에서는 락을 오래 잡지 않고, 항상 같은 순서로 락을 획득하도록 설계해 데드락을 줄이는 것이 중요합니다.

## 12. 낙관적 락과 비관적 락의 차이를 설명해주세요

### 예상 질문

> Optimistic Lock과 Pessimistic Lock의 차이는 무엇인가요?

### 핵심 답변

비관적 락은 충돌이 자주 발생한다고 보고 먼저 락을 잡습니다.

```sql
SELECT *
FROM product
WHERE id = 1
FOR UPDATE;
```

낙관적 락은 충돌이 드물다고 보고 먼저 락을 잡지 않습니다. 대신 업데이트 시점에 version을 비교합니다.

```sql
UPDATE product
SET stock = stock - 1,
    version = version + 1
WHERE id = 1
  AND version = 3;
```

| 구분 | 비관적 락 | 낙관적 락 |
|---|---|---|
| 관점 | 충돌이 자주 난다 | 충돌이 드물다 |
| 방식 | 먼저 잠금 | 나중에 version 검증 |
| 장점 | 정합성 강함 | 락 대기 적음 |
| 단점 | 대기, 데드락 가능 | 재시도 로직 필요 |

### 실무 사례

- 좌석 예매, 계좌 이체: 비관적 락 선호
- 게시글 수정, 사용자 프로필 수정: 낙관적 락 선호

### 면접 답변 예시

> 비관적 락은 충돌 가능성이 높다고 보고 데이터를 먼저 잠그는 방식이고, 낙관적 락은 충돌 가능성이 낮다고 보고 업데이트 시점에 version 등으로 충돌 여부를 확인하는 방식입니다. 비관적 락은 정합성은 강하지만 대기와 데드락 가능성이 있고, 낙관적 락은 성능상 유리하지만 충돌 시 재시도나 사용자 안내가 필요합니다.

---

## 13. 언제 낙관적 락을 사용하고, 언제 비관적 락을 사용하나요?

### 예상 질문

> 실무에서 어떤 기준으로 낙관적 락과 비관적 락을 선택하나요?

### 핵심 답변

선택 기준은 **충돌 빈도와 실패 비용**입니다.

충돌이 드물고 재시도해도 되는 경우는 낙관적 락이 적합합니다. 충돌이 잦고, 실패하면 금전적 손실이나 정합성 문제가 큰 경우는 비관적 락이 적합합니다.

### 낙관적 락이 적합한 경우

사용자 닉네임, 게시글 내용, 설정값 수정.

```text
동시에 수정할 가능성이 낮음
충돌 시 “다른 사용자가 먼저 수정했습니다” 안내 가능
```

### 비관적 락이 적합한 경우

콘서트 좌석 예매, 쿠폰 발급, 재고 차감.

```text
동시에 같은 자원에 접근할 가능성이 높음
초과 판매가 발생하면 비즈니스 손실 발생
```

### 면접 답변 예시

> 충돌이 적고 재시도가 가능한 도메인은 낙관적 락을 사용합니다. 반대로 재고 차감, 좌석 예매, 계좌 이체처럼 충돌 가능성이 높고 실패 비용이 큰 도메인은 비관적 락이나 조건부 `UPDATE`를 사용합니다. 중요한 건 락 자체보다 도메인의 실패 비용을 기준으로 판단하는 것입니다.

## 14. 분산 락이 무엇이고, 왜 사용하나요?

### 예상 질문

> 분산 락은 왜 필요한가요?

### 핵심 답변

분산 락은 여러 서버 인스턴스가 동시에 같은 공유 자원에 접근할 때, 하나의 서버만 특정 작업을 수행하도록 제어하는 락입니다.

단일 DB 안에서는 DB 락으로 충분할 수 있지만, 서버가 여러 대이고 작업 대상이 DB 외부 자원까지 포함되면 애플리케이션 레벨의 분산 락이 필요할 수 있습니다.

### 실무 사례

매일 새벽 1시에 정산 배치를 실행하는 서버가 5대라고 가정합니다. 각 서버가 스케줄러를 가지고 있으면 동일한 정산 작업이 5번 실행될 수 있습니다.

이때 Redis 분산 락을 사용해 한 서버만 작업하도록 합니다.

```text
서버 A: lock 획득 → 정산 실행
서버 B: lock 실패 → 실행하지 않음
서버 C: lock 실패 → 실행하지 않음
```

### 주의점

분산 락은 만능이 아닙니다.

- TTL 설정 필요
- 락 해제 실패 대비 필요
- 작업 시간이 TTL보다 길어질 때 대비 필요
- Redis 장애 시 정책 필요
- 가능하면 DB unique constraint나 idempotency key로도 방어 필요

### 면접 답변 예시

> 분산 락은 여러 애플리케이션 인스턴스가 같은 자원이나 작업을 동시에 처리하지 않도록 제어하는 락입니다. 예를 들어 여러 서버에서 같은 배치가 동시에 실행되는 것을 막을 때 사용할 수 있습니다. 다만 TTL, 락 해제, 장애 상황을 잘 설계해야 하고, 중요한 정합성은 DB 제약 조건이나 멱등성 키로도 한 번 더 방어해야 합니다.

## 15. Lost Update Problem이 무엇이고, 왜 트랜잭션 격리 수준으로 해결하지 못하나요?

### 예상 질문

> Lost Update가 무엇인가요?

### 핵심 답변

Lost Update는 두 트랜잭션이 같은 데이터를 읽고 각각 수정했을 때, 한쪽 수정 결과가 사라지는 문제입니다.

```text
초기 재고: 10

T1: 재고 10 읽음
T2: 재고 10 읽음
T1: 재고 9로 update
T2: 재고 9로 update

결과: 실제로 2개 팔렸지만 재고는 9
```

트랜잭션 격리 수준이 일정 수준 이상이면 일부 DB에서는 방지되지만, 실무에서는 격리 수준만 믿기 어렵습니다. ORM의 read-modify-write 패턴, DBMS별 구현 차이, 애플리케이션 로직 때문에 명시적 제어가 필요합니다.

### 실무 사례

```java
Product product = productRepository.findById(id);
product.decreaseStock(1);
```

이런 ORM 코드가 동시에 실행되면 Lost Update가 발생할 수 있습니다.

### 면접 답변 예시

> Lost Update는 두 트랜잭션이 같은 값을 읽고 각각 수정하면서 먼저 반영된 수정이 뒤의 수정에 덮어써지는 문제입니다. DBMS와 격리 수준에 따라 일부 방지될 수 있지만, 실무에서는 ORM의 read-modify-write 패턴 때문에 여전히 위험할 수 있습니다. 그래서 명시적으로 원자적 `UPDATE`, 비관적 락, 낙관적 락을 사용해 방어합니다.

## 16. Lost Update Problem을 해결하는 방법을 알려주세요

### 예상 질문

> Lost Update는 어떻게 해결하나요?

### 핵심 답변

대표적인 해결 방법은 네 가지입니다.

1. **원자적 UPDATE**
2. **비관적 락**
3. **낙관적 락**
4. **유니크 제약 또는 조건부 제약**

### 방법 1. 원자적 UPDATE

```sql
UPDATE product
SET stock = stock - 1
WHERE id = 1
  AND stock > 0;
```

가장 단순하고 강력한 방식입니다.

### 방법 2. 비관적 락

```sql
SELECT *
FROM product
WHERE id = 1
FOR UPDATE;
```

row를 먼저 잠근 뒤 수정합니다.

### 방법 3. 낙관적 락

```sql
UPDATE product
SET stock = 9,
    version = version + 1
WHERE id = 1
  AND version = 3;
```

업데이트 실패 시 재시도하거나 사용자에게 충돌을 알립니다.

### 실무 사례

재고 차감은 보통 다음 순서로 봅니다.

```text
1순위: 조건부 UPDATE
2순위: 비관적 락
3순위: 낙관적 락
4순위: 큐 기반 직렬화
```

### 면접 답변 예시

> 재고 차감처럼 숫자를 감소시키는 작업은 가능하면 DB의 원자적 `UPDATE`로 해결합니다. 복잡한 검증이 필요하면 `SELECT FOR UPDATE`로 비관적 락을 사용하고, 충돌이 적은 수정 작업은 version 컬럼을 이용한 낙관적 락을 사용합니다.

## 17. 특별 부록: 빈출 질문 PDF

### 추가 예상 질문

- 데드락은 왜 발생하나요?
- 데드락을 줄이는 방법은 무엇인가요?
- `SELECT FOR UPDATE`는 언제 사용하나요?
- Redis 분산 락만으로 정합성을 보장할 수 있나요?
- 락을 오래 잡으면 어떤 문제가 생기나요?

### 실무 사례

데드락은 보통 락 획득 순서가 다를 때 발생합니다.

```text
T1: account 1 lock → account 2 lock 대기
T2: account 2 lock → account 1 lock 대기
```

해결 방법은 항상 같은 순서로 락을 잡는 것입니다.

```sql
SELECT *
FROM account
WHERE id IN (1, 2)
ORDER BY id
FOR UPDATE;
```

### 면접 답변 예시

> 데드락은 여러 트랜잭션이 서로 상대방이 잡은 락을 기다리면서 발생합니다. 실무에서는 락 획득 순서를 통일하고, 트랜잭션을 짧게 유지하며, 대량 업데이트를 작은 단위로 나누어 처리해 데드락 가능성을 줄입니다.

## 18. 인덱스가 무엇이고 왜 사용하나요?

### 예상 질문

> 인덱스란 무엇인가요?

### 핵심 답변

인덱스는 테이블의 특정 컬럼을 기준으로 데이터를 빠르게 찾기 위한 자료구조입니다. 책의 목차나 색인과 비슷합니다.

인덱스가 없으면 DB는 조건에 맞는 데이터를 찾기 위해 테이블 전체를 스캔해야 합니다. 이를 Full Table Scan이라고 합니다. 인덱스가 있으면 B-Tree 같은 자료구조를 통해 필요한 범위만 빠르게 탐색할 수 있습니다.

### 실무 사례

```sql
SELECT *
FROM orders
WHERE user_id = 10
ORDER BY created_at DESC
LIMIT 20;
```

이 쿼리가 자주 실행된다면 다음 인덱스가 효과적일 수 있습니다.

```sql
CREATE INDEX idx_orders_user_created
ON orders(user_id, created_at DESC);
```

### 면접 답변 예시

> 인덱스는 조회 성능을 높이기 위한 보조 자료구조입니다. `WHERE`, `JOIN`, `ORDER BY`, `GROUP BY`에서 자주 사용되는 컬럼에 인덱스를 만들면 불필요한 전체 스캔을 줄일 수 있습니다. 다만 인덱스는 쓰기 성능과 저장 공간 비용이 있기 때문에 무조건 많이 만드는 것은 좋지 않습니다.

---

## 19. 인덱스는 왜 쓰기 성능을 떨어뜨릴 수 있나요?

### 예상 질문

> 인덱스를 많이 만들면 왜 안 좋나요?

### 핵심 답변

인덱스는 조회를 빠르게 하지만, `INSERT`, `UPDATE`, `DELETE` 시에는 인덱스도 함께 갱신해야 합니다. 즉, 데이터 row만 바꾸는 게 아니라 관련 인덱스 구조도 변경해야 합니다.

### 실무 사례

회원 테이블에 다음 인덱스가 모두 있다고 가정합니다.

```text
email
phone
nickname
created_at
last_login_at
status
```

회원 가입 `INSERT` 한 번이 발생할 때, 실제 row 추가뿐 아니라 여러 인덱스에도 값이 추가됩니다. 쓰기 트래픽이 많은 테이블에서는 인덱스가 많을수록 쓰기 비용이 커집니다.

### 면접 답변 예시

> 인덱스는 조회 성능을 위한 자료구조이지만, 데이터 변경 시 인덱스도 함께 갱신해야 합니다. 그래서 쓰기 트래픽이 많은 테이블에 인덱스를 과도하게 만들면 `INSERT`, `UPDATE`, `DELETE` 성능이 떨어지고 저장 공간도 증가합니다.

---

## 20. B-Tree 인덱스는 어떻게 동작하나요?

### 예상 질문

> B-Tree 인덱스의 동작 원리를 설명해주세요.

### 핵심 답변

B-Tree는 정렬된 상태를 유지하는 트리 기반 자료구조입니다. 루트 노드에서 시작해 중간 노드를 거쳐 리프 노드까지 내려가면서 원하는 값을 찾습니다.

B-Tree 인덱스는 다음에 강합니다.

- 동등 조건: `WHERE id = 10`
- 범위 조건: `WHERE created_at BETWEEN ...`
- 정렬: `ORDER BY created_at`
- prefix 기반 복합 인덱스 탐색

### 실무 사례

```sql
SELECT *
FROM payment
WHERE created_at >= '2026-04-01'
  AND created_at < '2026-05-01';
```

`created_at`에 B-Tree 인덱스가 있으면 특정 날짜 범위에 해당하는 구간만 스캔할 수 있습니다.

### 면접 답변 예시

> B-Tree 인덱스는 정렬된 트리 구조를 통해 루트에서 리프까지 탐색하면서 원하는 데이터를 찾는 자료구조입니다. 데이터가 정렬되어 있기 때문에 단건 조회뿐 아니라 범위 검색과 정렬에도 유리합니다.

---

## 21. 왜 Hash Table이 아니라 B-Tree를 사용하나요?

### 예상 질문

> 인덱스 자료구조로 왜 Hash Table이 아니라 B-Tree를 많이 사용하나요?

### 핵심 답변

Hash Table은 동등 비교에는 빠릅니다.

```sql
WHERE id = 10
```

하지만 범위 검색과 정렬에는 약합니다.

```sql
WHERE created_at >= '2026-04-01'
ORDER BY created_at DESC
```

반면 B-Tree는 데이터가 정렬된 구조로 유지되기 때문에 동등 검색뿐 아니라 범위 검색, 정렬, prefix 검색에 유리합니다.

### 실무 사례

커머스 주문 목록에서 사용자는 보통 다음처럼 조회합니다.

```sql
SELECT *
FROM orders
WHERE user_id = 10
  AND created_at >= '2026-04-01'
ORDER BY created_at DESC;
```

이런 쿼리는 단순 Hash보다 B-Tree 기반 복합 인덱스가 훨씬 적합합니다.

### 면접 답변 예시

> Hash Table은 `=` 조건에는 매우 빠르지만 범위 검색, 정렬, prefix 검색에 적합하지 않습니다. 반면 B-Tree는 정렬된 구조를 유지하기 때문에 `=`, 범위 조건, `ORDER BY`를 모두 지원하기 좋습니다. 실제 서비스 쿼리는 단건 조회뿐 아니라 기간 조회와 정렬이 많기 때문에 B-Tree가 범용적으로 많이 사용됩니다.

---

## 22. 복합 인덱스의 컬럼 순서는 어떻게 정하나요?

### 예상 질문

> 복합 인덱스에서 컬럼 순서는 어떻게 정해야 하나요?

### 핵심 답변

복합 인덱스는 컬럼 순서가 매우 중요합니다. 일반적으로 다음 기준을 봅니다.

1. 동등 조건 컬럼
2. 선택도가 높은 컬럼
3. 범위 조건 컬럼
4. 정렬에 사용되는 컬럼

예를 들어 다음 쿼리가 많다면,

```sql
SELECT *
FROM orders
WHERE user_id = 10
  AND status = 'PAID'
ORDER BY created_at DESC;
```

다음 인덱스를 고려할 수 있습니다.

```sql
CREATE INDEX idx_orders_user_status_created
ON orders(user_id, status, created_at DESC);
```

### 주의점

`(user_id, status, created_at)` 인덱스는 `user_id` 없이 `status`만으로는 효율적으로 사용되기 어렵습니다. 이를 **leftmost prefix rule**이라고 설명할 수 있습니다.

### 면접 답변 예시

> 복합 인덱스는 컬럼 순서가 중요합니다. 보통 `WHERE`의 동등 조건 컬럼을 앞에 두고, 그다음 범위 조건이나 정렬 컬럼을 배치합니다. 또한 복합 인덱스는 왼쪽부터 사용되기 때문에, 앞 컬럼을 건너뛰면 인덱스를 효율적으로 사용하기 어렵습니다.

---

## 23. 인덱스를 타지 않는 경우는 어떤 경우인가요?

### 예상 질문

> 인덱스를 만들었는데도 쿼리가 인덱스를 안 타는 이유는 무엇인가요?

### 핵심 답변

대표적인 경우는 다음과 같습니다.

- 인덱스 컬럼에 함수를 적용한 경우
- 앞쪽 와일드카드 `LIKE`를 사용한 경우
- 타입이 맞지 않아 형변환이 발생한 경우
- 선택도가 낮은 컬럼인 경우
- 복합 인덱스의 앞 컬럼을 사용하지 않은 경우
- 통계 정보가 부정확한 경우

### 예시

```sql
SELECT *
FROM users
WHERE DATE(created_at) = '2026-04-28';
```

이 경우 `created_at`에 함수가 적용되어 인덱스를 제대로 활용하기 어렵습니다.

개선하면 다음과 같습니다.

```sql
SELECT *
FROM users
WHERE created_at >= '2026-04-28 00:00:00'
  AND created_at < '2026-04-29 00:00:00';
```

### 면접 답변 예시

> 인덱스를 생성해도 컬럼에 함수를 적용하거나, 타입 변환이 발생하거나, 복합 인덱스의 앞 컬럼을 사용하지 않으면 인덱스를 제대로 못 탈 수 있습니다. 그래서 실행 계획을 확인하고, 조건식을 인덱스 컬럼 그대로 비교하도록 바꾸는 것이 중요합니다.

---

## 24. 커버링 인덱스란 무엇인가요?

### 예상 질문

> 커버링 인덱스가 무엇인가요?

### 핵심 답변

커버링 인덱스는 쿼리에 필요한 컬럼이 모두 인덱스에 포함되어 있어서, 테이블 row를 추가로 조회하지 않아도 되는 인덱스입니다.

### 실무 사례

```sql
SELECT user_id, created_at
FROM orders
WHERE status = 'PAID'
ORDER BY created_at DESC
LIMIT 20;
```

다음 인덱스가 있다면,

```sql
CREATE INDEX idx_orders_status_created_user
ON orders(status, created_at DESC, user_id);
```

DB는 인덱스만 보고 결과를 만들 수 있습니다.

### 면접 답변 예시

> 커버링 인덱스는 쿼리에 필요한 컬럼을 인덱스가 모두 포함해서 테이블 접근을 줄이는 방식입니다. 조회 성능은 좋아지지만 인덱스 크기가 커지고 쓰기 비용이 증가하므로, 자주 호출되는 핵심 조회 쿼리에 제한적으로 적용하는 것이 좋습니다.

---

## 25. 인덱스를 운영 중인 대용량 테이블에 추가할 때 주의할 점은?

### 예상 질문

> 운영 중인 테이블에 인덱스를 추가할 때 무엇을 고려해야 하나요?

### 핵심 답변

대용량 테이블에 인덱스를 추가하면 CPU, I/O, 락, 복제 지연, 배포 시간 문제가 생길 수 있습니다.

PostgreSQL에서는 `CREATE INDEX CONCURRENTLY`를 사용하면 일반 인덱스 생성보다 쓰기 차단을 줄일 수 있지만, 더 오래 걸리고 실패 시 invalid index 정리가 필요할 수 있습니다.

참고: <https://www.postgresql.org/docs/current/sql-createindex.html>

### 실무 체크리스트

- 트래픽 적은 시간에 실행
- 복제 지연 모니터링
- CPU/I/O 모니터링
- 쿼리 플랜 사전 확인
- 롤백 계획 준비
- PostgreSQL이면 `CONCURRENTLY` 검토
- MySQL이면 online DDL 옵션 확인

### 실무 사례

1,800만 건 테이블에 인덱스를 그냥 추가했다가 CPU 100%와 서비스 지연이 발생할 수 있습니다. 운영 DB에서는 “인덱스 추가도 배포”로 보고 점진적으로 진행해야 합니다.

### 면접 답변 예시

> 운영 중 대용량 테이블에 인덱스를 추가할 때는 락, I/O, CPU, replication lag를 고려해야 합니다. 트래픽이 낮은 시간에 진행하고, DBMS별 online DDL 또는 concurrent index 생성 옵션을 검토합니다. 또한 실행 계획과 롤백 계획을 미리 준비해야 합니다.

---

## 26. 실행 계획을 어떻게 보고 인덱스 문제를 판단하나요?

### 예상 질문

> EXPLAIN을 보면 어떤 부분을 확인하나요?

### 핵심 답변

실행 계획에서는 다음을 봅니다.

- Full Table Scan인지 Index Scan인지
- 예상 row 수와 실제 row 수 차이
- 정렬이 filesort로 발생하는지
- 임시 테이블을 사용하는지
- JOIN 순서가 적절한지
- 사용된 인덱스가 기대한 인덱스인지

### 실무 사례

```sql
EXPLAIN
SELECT *
FROM orders
WHERE user_id = 10
ORDER BY created_at DESC
LIMIT 20;
```

여기서 `idx_orders_user_created`를 기대했는데 full scan이 나오면, 인덱스 컬럼 순서나 통계 정보, 조건식을 의심해야 합니다.

### 면접 답변 예시

> 실행 계획에서는 우선 어떤 접근 방식이 사용되는지 봅니다. Full scan인지, index scan인지, 예상 row 수가 과도하지 않은지, 정렬이나 임시 테이블이 발생하는지 확인합니다. 그리고 실제로 기대한 인덱스가 사용되는지 확인한 뒤, 조건식과 인덱스 컬럼 순서를 조정합니다.

## 27. RDBMS와 NoSQL의 차이를 설명해주세요

### 예상 질문

> RDBMS와 NoSQL은 어떤 차이가 있나요?

### 핵심 답변

RDBMS는 정해진 스키마와 관계, SQL, 트랜잭션 정합성을 중심으로 설계된 DB입니다. NoSQL은 문서형, 키-값, 컬럼형, 그래프형 등 다양한 모델이 있고, 유연한 스키마와 수평 확장성을 강점으로 합니다.

| 구분 | RDBMS | NoSQL |
|---|---|---|
| 데이터 모델 | 테이블/관계 | 문서, 키-값, 컬럼, 그래프 |
| 스키마 | 강함 | 유연함 |
| 트랜잭션 | 강함 | 제품별 차이 |
| 확장 | 수직 확장 중심, 샤딩 가능 | 수평 확장 친화 |
| 질의 | SQL | 제품별 API/쿼리 |

참고: <https://www.mongodb.com/docs/manual/>

### 실무 사례

- 결제, 정산, 계좌: RDBMS 적합
- 로그, 이벤트, 세션, 캐시성 데이터: NoSQL 적합
- 채팅 메시지, 피드: 요구사항에 따라 RDBMS/NoSQL 모두 가능

### 면접 답변 예시

> RDBMS는 정해진 스키마, 관계, 트랜잭션 정합성에 강하고, NoSQL은 유연한 스키마와 수평 확장에 강합니다. 저는 결제나 정산처럼 정합성이 중요한 코어 데이터는 RDBMS를 우선 고려하고, 로그나 캐시, 대량 이벤트처럼 확장성과 처리량이 중요한 데이터는 NoSQL을 고려합니다.

---

## 28. 언제 RDBMS를 사용하고 언제 NoSQL을 사용하나요?

### 예상 질문

> 서비스 설계에서 RDBMS와 NoSQL을 어떻게 선택하나요?

### 핵심 답변

선택 기준은 데이터의 성격입니다.

RDBMS는 다음에 적합합니다.

- 데이터 정합성이 중요
- 관계가 복잡함
- 트랜잭션이 중요
- SQL 분석/조인이 필요

NoSQL은 다음에 적합합니다.

- 스키마가 자주 바뀜
- 대량 쓰기/읽기가 필요
- 수평 확장이 중요
- 데이터 접근 패턴이 단순하고 명확함

### 실무 사례

커머스 시스템에서 주문, 결제, 환불은 RDBMS에 저장하고, 상품 조회 캐시나 사용자 행동 로그는 Redis, MongoDB, Elasticsearch, ClickHouse 같은 별도 저장소에 둘 수 있습니다.

### 면접 답변 예시

> 저는 기본적으로 정합성이 중요한 코어 도메인은 RDBMS를 먼저 선택합니다. NoSQL은 스키마 유연성, 대량 쓰기, 수평 확장, 조회 패턴 최적화가 필요한 경우에 보조 저장소 또는 특정 도메인 저장소로 선택합니다.

---

## 29. NoSQL은 트랜잭션을 지원하지 않나요?

### 예상 질문

> NoSQL은 트랜잭션이 없나요?

### 핵심 답변

예전에는 NoSQL을 “트랜잭션이 약한 DB”로 설명하는 경우가 많았지만, 지금은 제품마다 다릅니다. MongoDB도 다중 문서 트랜잭션을 지원합니다. 다만 NoSQL을 사용할 때는 RDBMS처럼 조인과 트랜잭션을 자유롭게 쓰는 모델보다는, 접근 패턴에 맞춰 데이터를 설계하는 것이 중요합니다.

참고: <https://www.mongodb.com/docs/manual/core/transactions/>

### 실무 사례

채팅 메시지는 다음처럼 한 문서에 필요한 정보를 중복 저장할 수 있습니다.

```json
{
  "roomId": "r1",
  "messageId": "m1",
  "senderId": "u1",
  "senderName": "김우승",
  "content": "안녕하세요",
  "createdAt": "2026-04-28T10:00:00"
}
```

`senderName`이 중복되지만, 메시지 조회 시 user 테이블과 매번 조인하지 않아도 됩니다.

### 면접 답변 예시

> NoSQL이 무조건 트랜잭션을 지원하지 않는 것은 아닙니다. MongoDB처럼 다중 문서 트랜잭션을 지원하는 제품도 있습니다. 다만 NoSQL은 보통 접근 패턴 중심으로 데이터를 설계하고, 조인과 트랜잭션에 의존하기보다 비정규화와 중복 저장을 활용하는 경우가 많습니다.

---

## 30. NoSQL에서 정규화보다 비정규화를 많이 하는 이유는?

### 예상 질문

> NoSQL에서는 왜 데이터를 중복해서 저장하나요?

### 핵심 답변

NoSQL은 조인보다 조회 패턴에 맞춘 데이터 모델링을 중시합니다. 읽기 성능을 높이기 위해 필요한 데이터를 한 문서나 한 파티션에 중복 저장하는 경우가 많습니다.

### 실무 사례

주문 조회 화면에서 매번 상품 테이블을 조인하지 않기 위해 주문 항목에 상품명, 가격, 썸네일을 스냅샷으로 저장합니다.

```json
{
  "orderId": 1,
  "items": [
    {
      "productId": 10,
      "productName": "키보드",
      "price": 120000
    }
  ]
}
```

이 방식은 상품명이 나중에 바뀌어도 주문 당시의 상품 정보를 보존할 수 있다는 장점도 있습니다.

### 면접 답변 예시

> NoSQL에서는 조인 비용을 줄이고 조회 성능을 높이기 위해 비정규화를 자주 사용합니다. 접근 패턴이 명확하다면 필요한 데이터를 한 문서나 파티션에 함께 저장해서 빠르게 읽을 수 있습니다. 다만 중복 데이터가 생기기 때문에 변경 동기화와 정합성 관리가 중요합니다.

---

## 31. CAP 이론을 설명해주세요

### 예상 질문

> CAP 이론이 무엇인가요?

### 핵심 답변

CAP 이론은 분산 시스템에서 다음 세 가지를 동시에 완벽히 만족할 수 없다는 개념입니다.

| 요소 | 의미 |
|---|---|
| Consistency | 모든 노드가 같은 데이터를 봄 |
| Availability | 모든 요청에 응답 가능 |
| Partition Tolerance | 네트워크 분할 상황에서도 동작 |

분산 시스템에서는 네트워크 장애를 피할 수 없기 때문에 Partition Tolerance를 전제로 두고, Consistency와 Availability 사이에서 선택해야 하는 경우가 많습니다.

### 실무 사례

채팅 서비스에서 일시적으로 읽음 표시가 늦게 반영되는 것은 허용할 수 있습니다. 하지만 결제 승인 상태가 서버마다 다르게 보이는 것은 위험합니다.

그래서 읽음 표시, 접속 상태, 추천 피드는 eventual consistency를 허용할 수 있지만, 결제와 정산은 강한 일관성이 필요합니다.

### 면접 답변 예시

> CAP 이론은 분산 시스템에서 일관성, 가용성, 파티션 허용성을 동시에 완벽히 만족하기 어렵다는 이론입니다. 네트워크 분할은 현실적으로 피할 수 없기 때문에, 장애 상황에서 일관성을 더 우선할지 가용성을 더 우선할지를 도메인 요구사항에 따라 선택해야 합니다.

---

## 32. Eventual Consistency란 무엇인가요?

### 예상 질문

> 최종적 일관성이 무엇인가요?

### 핵심 답변

Eventual Consistency는 데이터 변경 직후에는 모든 노드가 같은 값을 보지 못할 수 있지만, 시간이 지나면 결국 같은 상태로 수렴한다는 개념입니다.

### 실무 사례

사용자가 프로필 사진을 변경했는데, 어떤 화면에서는 바로 바뀌고 어떤 화면에서는 몇 초 뒤에 바뀔 수 있습니다. 이 정도는 비즈니스적으로 허용 가능합니다.

반면 계좌 잔액이나 결제 상태는 최종적 일관성만으로 처리하면 위험합니다.

### 면접 답변 예시

> 최종적 일관성은 즉시 모든 노드가 같은 값을 보는 것은 아니지만, 추가 변경이 없다면 결국 같은 상태로 수렴하는 모델입니다. 피드, 좋아요 수, 조회수, 알림 배지에는 사용할 수 있지만, 결제나 재고처럼 강한 정합성이 필요한 영역에는 신중해야 합니다.

---

## 33. Redis는 RDBMS를 대체할 수 있나요?

### 예상 질문

> Redis를 DB처럼 사용해도 되나요?

### 핵심 답변

Redis는 인메모리 기반 key-value 저장소로 매우 빠르지만, 일반적인 RDBMS를 완전히 대체하는 용도로 쓰기에는 주의가 필요합니다. 보통 캐시, 세션, 분산 락, rate limit, 랭킹, pub/sub 등에 사용합니다.

### 실무 사례

상품 상세 조회가 매우 많다면 다음처럼 사용할 수 있습니다.

```text
1. Redis에서 product:10 조회
2. 있으면 바로 반환
3. 없으면 DB 조회
4. Redis에 TTL과 함께 저장
```

하지만 주문, 결제, 정산 같은 원장성 데이터는 RDBMS에 저장하고 Redis는 캐시로 두는 것이 안전합니다.

### 면접 답변 예시

> Redis는 매우 빠른 인메모리 저장소지만, 일반적으로 RDBMS를 완전히 대체하기보다는 캐시, 세션, 분산 락, 랭킹 같은 용도로 사용합니다. 주문이나 결제처럼 원장성 데이터는 RDBMS에 저장하고, Redis는 조회 성능 개선을 위한 보조 저장소로 사용하는 것이 안전합니다.

---

## 34. Elasticsearch나 검색 엔진은 왜 별도로 쓰나요?

### 예상 질문

> RDBMS에도 LIKE 검색이 있는데 왜 Elasticsearch를 쓰나요?

### 핵심 답변

RDBMS의 `LIKE` 검색은 단순 검색에는 가능하지만, 전문 검색, 형태소 분석, 랭킹, 오타 허용, 복합 필터링에는 한계가 있습니다. 검색 엔진은 역색인 기반으로 대량 텍스트 검색에 최적화되어 있습니다.

### 실무 사례

쇼핑몰에서 “나이키 검정 운동화”를 검색할 때, 단순 `LIKE`보다 검색 엔진이 더 적합합니다.

다만 검색 엔진은 보조 저장소로 보는 것이 안전합니다. 원본 데이터는 RDBMS에 두고, 검색용 색인을 별도로 동기화하는 구조가 일반적입니다.

### 면접 답변 예시

> RDBMS의 `LIKE` 검색은 단순 문자열 검색에는 사용할 수 있지만, 대량 텍스트의 전문 검색, 랭킹, 형태소 분석, 오타 허용 같은 기능에는 한계가 있습니다. Elasticsearch 같은 검색 엔진은 역색인을 기반으로 검색에 최적화되어 있기 때문에 검색 전용 저장소로 사용합니다. 다만 원본 데이터는 별도 DB에 두고, 검색 엔진은 색인 저장소로 보는 것이 안전합니다.

# 섹션 6. 커넥션과 커넥션 풀

> 공개 페이지에서는 섹션 6이 “빈출 유형 5 - 커넥션과 커넥션 풀”이고 6개 강의로 구성된 것까지 확인됩니다.

---

## 35. DB 커넥션이란 무엇인가요?

### 예상 질문

> DB 커넥션이 무엇인가요?

### 핵심 답변

DB 커넥션은 애플리케이션과 DB 서버 사이의 연결입니다. 애플리케이션은 이 연결을 통해 SQL을 보내고 결과를 받습니다.

커넥션을 매 요청마다 새로 만들면 TCP 연결, 인증, 세션 초기화 비용이 발생합니다. 그래서 보통 커넥션 풀을 사용합니다.

### 실무 사례

비효율적인 구조:

```text
요청 1 → 커넥션 생성 → SQL 실행 → 커넥션 종료
요청 2 → 커넥션 생성 → SQL 실행 → 커넥션 종료
```

일반적인 구조:

```text
서버 시작 시 커넥션 풀 생성
요청 → 커넥션 빌림 → SQL 실행 → 커넥션 반납
```

### 면접 답변 예시

> DB 커넥션은 애플리케이션과 DB 사이의 통신 연결입니다. 매 요청마다 새로 생성하면 연결 생성, 인증, 초기화 비용이 크기 때문에 보통 커넥션 풀을 사용해서 미리 만든 커넥션을 재사용합니다.

---

## 36. 커넥션 풀이란 무엇이고 왜 사용하나요?

### 예상 질문

> 커넥션 풀은 왜 필요한가요?

### 핵심 답변

커넥션 풀은 DB 커넥션을 미리 만들어두고 재사용하는 구조입니다. 매번 커넥션을 생성하고 종료하는 비용을 줄이고, DB에 동시에 접근하는 커넥션 수를 제한할 수 있습니다.

Java/Spring 환경에서는 HikariCP가 널리 사용됩니다.

참고: <https://github.com/brettwooldridge/HikariCP>

### 실무 사례

Spring Boot + HikariCP에서 `maximumPoolSize = 10`이면 동시에 DB 작업을 수행할 수 있는 커넥션은 최대 10개입니다. 11번째 요청은 커넥션이 반납될 때까지 기다립니다.

### 면접 답변 예시

> 커넥션 풀은 DB 커넥션을 미리 만들어두고 재사용하는 구조입니다. 연결 생성 비용을 줄이고, 동시에 DB에 접근하는 커넥션 수를 제한해서 DB를 보호할 수 있습니다. 다만 풀 크기를 무조건 크게 잡으면 DB 부하가 커질 수 있으므로 적절한 사이징이 필요합니다.

---

## 37. 커넥션 풀이 고갈되면 어떤 일이 발생하나요?

### 예상 질문

> 커넥션 풀 고갈은 왜 발생하고 어떻게 확인하나요?

### 핵심 답변

커넥션 풀이 고갈되면 애플리케이션은 DB 커넥션을 빌리지 못해 요청이 대기하다가 타임아웃됩니다.

원인은 다양합니다.

- 느린 쿼리
- 긴 트랜잭션
- 커넥션 반납 누락
- 외부 API 호출을 트랜잭션 내부에서 수행
- DB 장애
- 트래픽 급증
- 풀 사이즈 과소 설정

### 실무 사례

```text
maximumPoolSize = 10
느린 쿼리 10개가 각각 30초 실행
→ 모든 커넥션 점유
→ 신규 요청은 대기
→ connection timeout 발생
```

### 면접 답변 예시

> 커넥션 풀 고갈이 발생하면 단순히 pool size를 늘리기보다 active connection 수, 대기 시간, slow query, 트랜잭션 시간을 먼저 봐야 합니다. 풀을 늘리면 일시적으로 완화될 수 있지만 DB 부하가 더 커질 수 있습니다.

---

## 38. 커넥션 풀 사이즈는 어떻게 정하나요?

### 예상 질문

> 커넥션 풀 크기는 크게 잡을수록 좋은가요?

### 핵심 답변

커넥션 풀은 크게 잡는다고 무조건 좋은 것이 아닙니다. 커넥션이 너무 많으면 DB 서버의 CPU, 메모리, 컨텍스트 스위칭 비용이 증가합니다.

기준은 다음과 같습니다.

- DB가 감당 가능한 최대 커넥션 수
- 애플리케이션 인스턴스 수
- 요청당 DB 사용 시간
- 평균/피크 TPS
- 쿼리 실행 시간
- read/write 비율

### 실무 사례

```text
DB max_connections = 300
애플리케이션 서버 = 10대
서버당 pool size = 50
총 가능 커넥션 = 500
```

이러면 DB max_connections를 초과할 수 있습니다.

현실적으로는 다음처럼 계산합니다.

```text
서버당 pool size = DB 허용 커넥션 / 앱 인스턴스 수 - 여유분
```

### 면접 답변 예시

> 커넥션 풀 사이즈는 DB의 최대 커넥션 수, 애플리케이션 인스턴스 수, 요청당 DB 점유 시간, 피크 트래픽을 고려해서 정합니다. 풀 크기를 무조건 크게 잡으면 DB가 감당하지 못할 수 있으므로, 전체 인스턴스의 총 커넥션 수가 DB 한도를 넘지 않도록 해야 합니다.

---

## 39. 커넥션 타임아웃과 쿼리 타임아웃의 차이는?

### 예상 질문

> connection timeout과 query timeout은 무엇이 다른가요?

### 핵심 답변

커넥션 타임아웃은 커넥션 풀에서 커넥션을 빌리는 데 기다릴 수 있는 최대 시간입니다.

쿼리 타임아웃은 커넥션을 얻은 뒤 SQL 실행이 허용되는 최대 시간입니다.

| 구분 | 의미 |
|---|---|
| Connection Timeout | 풀에서 커넥션을 빌리기까지 기다리는 시간 |
| Query Timeout | SQL 실행이 끝나기까지 기다리는 시간 |
| Socket Timeout | DB와 네트워크 통신 응답 대기 시간 |
| Idle Timeout | 사용하지 않는 커넥션 유지 시간 |
| Max Lifetime | 커넥션 최대 생존 시간 |

### 실무 사례

커넥션 풀은 여유가 있는데 특정 쿼리만 60초 걸린다면 query timeout 문제입니다. 반대로 모든 요청이 커넥션을 못 얻고 실패하면 connection pool 고갈 문제입니다.

### 면접 답변 예시

> connection timeout은 커넥션 풀에서 커넥션을 빌리기 위해 기다리는 시간이고, query timeout은 커넥션을 얻은 뒤 SQL 실행을 기다리는 시간입니다. 전자는 커넥션 풀 고갈과 관련이 있고, 후자는 느린 쿼리나 DB 처리 지연과 관련이 있습니다.

---

## 40. 커넥션 누수는 어떻게 찾나요?

### 예상 질문

> 커넥션 leak은 어떻게 탐지하고 해결하나요?

### 핵심 답변

커넥션 누수는 빌린 커넥션을 반납하지 않는 문제입니다. 보통 try-with-resources를 사용하지 않거나, 트랜잭션이 비정상적으로 종료되거나, 커넥션을 직접 다루는 코드에서 발생합니다.

### 확인 방법

- active connection이 계속 증가하는지 확인
- idle connection이 회복되지 않는지 확인
- HikariCP leak detection 설정
- thread dump 확인
- slow query log 확인
- 트랜잭션 경계 확인

### 실무 사례

문제 코드:

```java
Connection conn = dataSource.getConnection();
// 예외 발생
// conn.close() 호출 안 됨
```

개선 코드:

```java
try (Connection conn = dataSource.getConnection()) {
    // SQL 실행
}
```

Spring/JPA를 사용한다면 커넥션을 직접 다루기보다 트랜잭션 매니저에 맡기는 것이 안전합니다.

### 면접 답변 예시

> 커넥션 누수는 커넥션을 빌린 뒤 반납하지 않아 풀이 점점 고갈되는 문제입니다. active connection이 계속 증가하고 idle이 회복되지 않으면 의심할 수 있습니다. HikariCP의 leak detection, thread dump, 트랜잭션 경계를 확인하고, 직접 커넥션을 다루는 코드는 try-with-resources로 반납을 보장해야 합니다.

# 섹션 7. 운영

## 41. DB 장애가 발생하면 어떤 순서로 접근하나요?

### 예상 질문

> 운영 중 DB 장애가 발생하면 어떤 순서로 대응하나요?

### 핵심 답변

장애 대응은 원인 분석보다 **영향 축소가 먼저**입니다.

순서는 다음과 같습니다.

1. 장애 범위 확인
2. 사용자 영향 확인
3. DB 연결 가능 여부 확인
4. 애플리케이션 에러율 확인
5. slow query / lock / connection pool 확인
6. 최근 배포·DDL·배치 확인
7. 임시 완화 조치
8. 근본 원인 분석
9. 재발 방지

### 실무 사례

```text
증상: API 500 증가
확인: DB CPU 100%, active connection 급증
원인 후보: 새로 배포한 쿼리가 full scan
임시 조치: 해당 API feature flag off
후속 조치: 인덱스 추가, 쿼리 수정, 부하 테스트
```

### 면접 답변 예시

> 장애가 나면 먼저 사용자의 영향과 장애 범위를 확인합니다. 그다음 DB CPU, 커넥션 풀, slow query, lock wait, 최근 배포나 배치 여부를 봅니다. 원인 분석과 동시에 트래픽 차단, 기능 플래그 off, read replica 우회, 배치 중단 같은 완화 조치를 먼저 수행합니다.

## 42. Slow Query가 발생하면 어떻게 분석하나요?

### 예상 질문

> 느린 쿼리는 어떻게 튜닝하나요?

### 핵심 답변

순서는 다음과 같습니다.

1. slow query log 확인
2. 실행 빈도 확인
3. 실행 계획 확인
4. 인덱스 사용 여부 확인
5. 읽는 row 수 확인
6. 정렬/임시 테이블 발생 여부 확인
7. 쿼리 rewrite
8. 인덱스 추가 또는 변경
9. 캐시 적용 검토

### 실무 사례

```sql
SELECT *
FROM orders
WHERE DATE(created_at) = '2026-04-28';
```

이 쿼리는 인덱스를 못 탈 수 있습니다.

개선:

```sql
SELECT *
FROM orders
WHERE created_at >= '2026-04-28 00:00:00'
  AND created_at < '2026-04-29 00:00:00';
```

### 면접 답변 예시

> Slow Query가 발생하면 먼저 slow query log로 어떤 쿼리가 얼마나 자주 느린지 확인합니다. 그다음 `EXPLAIN`으로 실행 계획을 보고 full scan, filesort, 임시 테이블, 인덱스 미사용 여부를 확인합니다. 이후 조건식을 수정하거나 인덱스를 조정하고, 필요하면 캐시나 비동기 처리를 검토합니다.

## 43. DB CPU가 갑자기 100%가 되면 무엇을 확인하나요?

### 예상 질문

> DB CPU가 급증하면 어떻게 원인을 찾나요?

### 핵심 답변

확인 순서는 다음과 같습니다.

- 최근 배포 여부
- 갑자기 늘어난 쿼리
- full scan 쿼리
- 인덱스 미사용
- 정렬/집계 쿼리
- 배치 작업
- 커넥션 수 증가
- lock wait
- autovacuum 또는 background 작업

### 실무 사례

운영 중 인덱스 없는 컬럼으로 검색 기능이 배포되면, 트래픽이 많아지는 순간 DB CPU가 급증할 수 있습니다.

```sql
SELECT *
FROM users
WHERE phone_number = '010...';
```

`phone_number` 인덱스가 없다면 full scan이 발생할 수 있습니다.

### 면접 답변 예시

> DB CPU가 급증하면 최근 배포와 배치 작업을 먼저 확인하고, 현재 실행 중인 쿼리와 slow query를 봅니다. full scan, 정렬, 집계, 인덱스 미사용 쿼리가 있는지 실행 계획을 확인하고, 필요하면 해당 기능을 일시적으로 차단하거나 쿼리를 수정합니다.

## 44. DB 락 대기가 길어지면 어떻게 대응하나요?

### 예상 질문

> lock wait이 증가하면 어떻게 해결하나요?

### 핵심 답변

먼저 어떤 트랜잭션이 락을 잡고 있는지 확인해야 합니다.

확인할 것:

- 오래 열린 트랜잭션
- `SELECT FOR UPDATE`
- 대량 `UPDATE`/`DELETE`
- DDL 실행 여부
- 배치 작업
- 데드락 로그
- 트랜잭션 경계

### 실무 사례

관리자 페이지에서 다음 쿼리가 실행되었다고 가정합니다.

```sql
UPDATE orders
SET status = 'EXPIRED'
WHERE created_at < '2025-01-01';
```

수백만 건을 한 번에 업데이트하면 락과 replication lag가 발생할 수 있습니다.

개선:

```sql
UPDATE orders
SET status = 'EXPIRED'
WHERE id BETWEEN 1 AND 10000;
```

작은 단위로 나누어 처리합니다.

### 면접 답변 예시

> lock wait이 길어지면 먼저 어떤 세션이 락을 잡고 있고 어떤 세션이 대기 중인지 확인합니다. 오래 열린 트랜잭션, 대량 업데이트, DDL, 배치 작업을 우선 의심합니다. 임시로는 문제 세션 종료나 배치 중단을 고려하고, 근본적으로는 트랜잭션 범위 축소와 배치 단위 분할로 해결합니다.

## 45. Replication Lag가 발생하면 어떤 문제가 생기나요?

### 예상 질문

> DB replication lag가 발생하면 서비스에 어떤 영향이 있나요?

### 핵심 답변

Replication Lag는 primary DB의 변경이 replica DB에 늦게 반영되는 현상입니다.

문제는 다음과 같습니다.

- 방금 쓴 데이터를 읽지 못함
- 사용자에게 이전 상태가 보임
- 정산/통계 데이터 불일치
- read-after-write 문제 발생

### 실무 사례

사용자가 프로필을 수정한 직후 마이페이지를 조회했는데, 조회가 replica로 가면 이전 프로필이 보일 수 있습니다.

해결 방법:

- 쓰기 직후 일정 시간 primary에서 읽기
- 사용자 단위 read-your-write 보장
- lag가 큰 replica는 로드밸런싱 제외
- 중요한 조회는 primary 사용

### 면접 답변 예시

> Replication lag가 생기면 primary에 반영된 데이터가 replica에는 아직 없어서 방금 쓴 데이터를 읽지 못하는 문제가 발생할 수 있습니다. 프로필 수정 직후 이전 정보가 보이는 식입니다. 중요한 read-after-write 조회는 primary로 보내고, lag가 큰 replica는 읽기 트래픽에서 제외하는 방식으로 대응합니다.

## 46. DB 백업과 복구 전략은 어떻게 세우나요?

### 예상 질문

> DB 백업은 어떻게 설계해야 하나요?

### 핵심 답변

백업 전략은 단순히 백업 파일을 만드는 것이 아니라 **복구 가능성**을 보장하는 것입니다.

핵심 지표는 두 가지입니다.

| 지표 | 의미 |
|---|---|
| RPO | 얼마나 많은 데이터 손실을 허용할 수 있는가 |
| RTO | 얼마나 빨리 복구해야 하는가 |

### 실무 사례

결제 시스템은 RPO가 거의 0에 가까워야 합니다. 즉, 장애 시 데이터 손실을 거의 허용할 수 없습니다. 반면 로그 분석 데이터는 몇 분 정도 손실을 허용할 수 있습니다.

### 면접 답변 예시

> 백업은 생성보다 복구 테스트가 중요합니다. RPO와 RTO를 먼저 정의하고, 스냅샷, WAL/binlog, point-in-time recovery를 조합합니다. 또한 정기적으로 복구 리허설을 해서 실제 장애 시 복구 가능한지 검증해야 합니다.

## 47. DB 마이그레이션은 어떻게 안전하게 진행하나요?

### 예상 질문

> 운영 DB 스키마 변경은 어떻게 안전하게 하나요?

### 핵심 답변

운영 DB 마이그레이션은 애플리케이션 배포와 DB 변경의 호환성을 고려해야 합니다.

안전한 순서는 보통 다음과 같습니다.

```text
1. nullable 컬럼 추가
2. 애플리케이션에서 새 컬럼 쓰기 시작
3. 기존 데이터 backfill
4. 읽기 로직 전환
5. not null 제약 추가
6. 기존 컬럼 제거
```

### 실무 사례

`users.name`을 `first_name`, `last_name`으로 분리한다고 가정합니다.

나쁜 방식:

```text
한 번에 컬럼 삭제 + 코드 변경 + 배포
```

좋은 방식:

```text
새 컬럼 추가
→ dual write
→ backfill
→ read 전환
→ 기존 컬럼 제거
```

### 면접 답변 예시

> 운영 DB 마이그레이션은 애플리케이션 배포와 호환되도록 단계적으로 진행해야 합니다. 먼저 nullable 컬럼을 추가하고, 새 코드에서 dual write를 수행한 뒤, backfill을 진행하고, 읽기 로직을 전환합니다. 마지막에 제약 조건을 추가하거나 기존 컬럼을 제거하는 방식이 안전합니다.

## 48. 대용량 데이터 삭제는 어떻게 처리하나요?

### 예상 질문

> 수천만 건 데이터를 삭제해야 하면 어떻게 하나요?

### 핵심 답변

대용량 `DELETE`를 한 번에 실행하면 락, undo/redo 로그 증가, replication lag, 디스크 I/O 증가 문제가 발생할 수 있습니다.

### 실무 방식

- 작은 배치로 나누어 삭제
- sleep을 두어 DB 부하 조절
- 파티션을 사용했다면 partition drop 고려
- 삭제 전 archive 여부 확인
- replica lag 모니터링
- 인덱스 영향 확인

### 예시

```sql
DELETE FROM access_logs
WHERE created_at < '2025-01-01'
LIMIT 10000;
```

이를 반복 실행합니다.

더 나은 구조는 처음부터 날짜 기준 파티셔닝을 하고, 오래된 파티션을 drop하는 것입니다.

### 면접 답변 예시

> 대용량 삭제는 한 번에 처리하면 락, 로그 증가, replication lag가 커질 수 있기 때문에 작은 배치로 나누어 처리합니다. 중간중간 sleep을 두고 replica lag와 DB 부하를 모니터링합니다. 날짜 기준 데이터라면 파티셔닝을 적용해 오래된 파티션을 drop하는 방식이 더 효율적입니다.

## 49. DB 성능 문제를 예방하기 위해 어떤 모니터링을 하나요?

### 예상 질문

> 운영 DB에서 어떤 지표를 모니터링해야 하나요?

### 핵심 답변

DB 운영 모니터링은 CPU만 보면 부족합니다. 최소한 다음을 봐야 합니다.

| 영역 | 지표 |
|---|---|
| 리소스 | CPU, Memory, Disk I/O, Disk 사용량 |
| 커넥션 | active connection, idle connection, wait count |
| 쿼리 | slow query, QPS, TPS, 평균/상위 latency |
| 락 | lock wait, deadlock count |
| 복제 | replication lag |
| 트랜잭션 | long transaction |
| 스토리지 | table/index bloat, vacuum 상태 |
| 에러 | connection timeout, deadlock, too many connections |

### 실무 사례

API latency가 증가했는데 애플리케이션 CPU는 정상이라면 DB 지표를 봐야 합니다.

```text
API p95 latency 증가
→ DB active connection 증가
→ slow query 증가
→ 특정 검색 API full scan 확인
→ 인덱스 추가 또는 쿼리 수정
```

### 면접 답변 예시

> DB 모니터링은 CPU, 메모리뿐 아니라 slow query, lock wait, connection pool, replication lag, long transaction을 함께 봐야 합니다. 장애 예방 관점에서는 임계치 알림뿐 아니라 배포 직후 특정 쿼리의 latency와 실행 계획 변화도 확인합니다.

# 마지막 정리

이 커리큘럼에서 핵심은 단순히 “트랜잭션, 락, 인덱스, NoSQL, 커넥션 풀”의 정의를 외우는 것이 아닙니다. 면접관은 보통 다음을 확인합니다.
