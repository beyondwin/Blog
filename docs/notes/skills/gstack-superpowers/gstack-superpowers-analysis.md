# `GStack + Superpowers로 클로드코드 완전히 달라진 사람의 후기` 분석 문서

- 영상: [YouTube](https://www.youtube.com/watch?v=af3OJ0L1jEU)
- 채널: 메이커 에반 | Maker Evan
- 업로드일: 2026-04-15
- 길이: 9분 7초
- 분석 기준: 영상 메타데이터, 자동 생성 자막, 스토리보드 프레임

## 1. 한 줄 요약

이 영상에서 말하는 핵심 작업 방식은 `바로 만들기`가 아니라, `아이디어 정리 -> 문서화 -> 계획 수립 -> 작업 분리 -> 병렬 실행 -> 중간 리뷰` 순서로 Claude Code를 운용하는 것이다.

## 2. 발표자가 실제로 설명한 작업 흐름

영상에서 발표자가 반복해서 강조하는 흐름은 아래 순서다.

### 2-1. 시작은 바로 구현이 아니라 `Office Hours`

- 새 기능이나 프로젝트를 시작할 때 바로 구현 지시를 하지 않는다.
- 먼저 `Office Hours`를 실행해서 Claude와 짧은 인터뷰를 한다.
- 이 단계에서 아래 같은 질문으로 요구사항을 선명하게 만든다.
  - 왜 만들려고 하는가
  - 어떤 상황에서 쓰는가
  - 비슷한 도구의 불편함은 무엇이었는가
  - 팀 규모나 사용 환경은 어떤가

영상 근거:

- `4:16~5:08` 구간에서 “항상 오피스 아워스부터 시작한다”, “바로 만들어 달라고 하지 않는다”고 설명
- `1:43~2:37` 구간에서 Office Hours를 “상담 시간”, “아이디어 회의”라고 설명

의미:

- Claude가 임의 해석으로 결과물을 만들어 버리는 문제를 줄인다.
- 요구사항이 흐릿한 상태에서 생기는 “이게 아닌데” 문제를 초반에 잡는다.

### 2-2. 대화 내용을 바로 문서로 정리

- Office Hours가 끝나면 내용을 문서로 남긴다.
- 문서 항목은 대체로 아래 구조다.
  - 뭘 만들 것인가
  - 왜 만드는가
  - 어떻게 구성할 것인가

영상 근거:

- `4:39~4:46` 부근에서 “그 내용을 문서로 정리한다”고 직접 설명

의미:

- 머릿속 아이디어를 실행 기준 문서로 바꾼다.
- 이후 플랜 작성과 리뷰의 입력값이 된다.

### 2-3. `Brainstorming`으로 빠진 구멍 점검

- 문서화 뒤에 바로 구현하지 않고 `Brainstorming` 스킬로 한 번 더 점검한다.
- 이 단계는 “혹시 놓친 게 있나”, “방향이 맞나”를 확인하는 보강 단계다.

영상 근거:

- `4:47~5:06` 구간에서 Brainstorming을 “마무리 검토 단계”라고 설명
- `6:57~7:02` 부근에서도 같은 순서를 다시 반복

의미:

- 초반 설계의 빈 구멍을 구현 전에 찾는다.
- 발표자는 특히 UX/UI 아이디어를 빠르게 시각화하는 데 유용하다고 본다.

### 2-4. `Writing-Plans`로 구현 전에 설계도 작성

- 그 다음 단계가 `Writing-Plans`다.
- 발표자는 이를 “공사 전에 설계 도면 그리는 것”에 비유한다.
- 여기서 Claude가 단계별 구현 계획서를 만든다.

영상 근거:

- `5:13~5:54` 구간 전체
- `7:03~7:10` 부근에서 “대화 내용을 기반으로 단계별 계획서를 만들어 준다”고 재설명

이 계획서에 기대하는 역할:

- 작업 순서 고정
- 중간에 방향 상실 방지
- 막혔을 때 기준점 제공

### 2-5. 계획서는 `리뷰`까지 해서 이중 검증

- 발표자는 플랜이 나오면 그대로 쓰지 않고 리뷰 에이전트/리뷰 스킬로 다시 본다고 말한다.
- 즉, 계획도 초안으로 보고 검토를 한 번 더 거친다.

영상 근거:

- `7:12~7:18` 구간에서 “플랜에 대해 리뷰를 추가적으로 꼭 해 준다”, “이중 검증”이라고 설명

의미:

- 구현 전에 설계 오류를 줄인다.
- 코드 리뷰를 구현 후가 아니라 설계 단계까지 당겨온 셈이다.

### 2-6. `Git WorkTrees`로 작업 공간 분리

- 구현에 들어가면 `Git WorkTrees`로 작업 공간을 기능별/작업 단위별로 나눈다.
- 발표자는 같은 책상 위에 여러 프로젝트 서류를 섞어 두지 않는 것에 비유한다.

영상 근거:

- `3:13~4:08` 구간
- `7:20~7:30` 구간에서 다시 “기능별로 작업 공간을 분리한다”고 설명

기대 효과:

- 한 작업이 다른 작업 컨텍스트를 오염시키지 않음
- 병렬 작업 가능
- 되돌아가기와 기록 확인이 쉬움

### 2-7. `Subagent Driven Development`로 병렬 실행

- 작업 공간을 나눈 뒤에는 `Subagent Driven Development`로 큰 작업을 작은 단위로 쪼개 처리한다.
- 발표자는 이를 회사에서 기획, 디자인, 개발, QA를 나눠서 일하는 구조에 비유한다.

영상 근거:

- `2:37~3:13` 구간
- `7:31~7:37` 구간에서 실제 실행 단계로 다시 언급

기대 효과:

- 컨텍스트 분리
- 역할 집중
- 큰 작업에서 품질 저하 방지

### 2-8. 구현 중간에는 `Design Review`로 AI 슬롭 제거

- GStack의 두 번째 핵심 기능으로 `Design Review`를 소개한다.
- 발표자는 이를 “과도한 그라데이션, 과도한 이모지” 같은 AI 특유의 어색함을 잡아내는 역할이라고 설명한다.

영상 근거:

- `2:20~2:37` 구간
- 마지막 요약 `8:23~8:29` 부근에서도 같은 의미로 정리

의미:

- 구현 자체보다 결과물의 완성도와 취향 보정을 담당한다.
- 바이브 코딩 냄새를 줄이는 품질 보정 단계다.

### 2-9. 자동 기록 기능을 작업 흐름의 일부로 사용

- 발표자는 Superpowers의 장점으로 자동 작업 기록을 꽤 높게 평가한다.
- “의미 있는 작업 단위가 될 때마다 알아서 기록을 남긴다”고 설명한다.

영상 근거:

- `3:45~4:07` 구간
- `7:27~7:30` 구간

의미:

- 흐름을 끊지 않고 히스토리를 남긴다.
- “언제 무엇이 바뀌었는지”를 나중에 다시 찾기 쉬워진다.

## 3. 이 영상을 기준으로 재구성한 실제 작업 SOP

발표자의 설명을 실행 절차로 다시 쓰면 아래와 같다.

1. 새 작업이 생기면 바로 구현하지 않는다.
2. `Office Hours`로 요구사항 인터뷰를 돌린다.
3. 대화 결과를 짧은 기획 문서로 정리한다.
4. `Brainstorming`으로 빠진 요구사항과 UX/UI 구멍을 찾는다.
5. `Writing-Plans`로 단계별 실행 계획서를 만든다.
6. 계획서에 대해 리뷰를 한 번 더 태운다.
7. `Git WorkTrees`로 작업 단위를 분리한다.
8. `Subagent Driven Development`로 각 단위를 병렬 실행한다.
9. 중간중간 `Design Review`로 어색한 산출물을 걷어낸다.
10. 자동 기록을 활용해 작업 단위별 변경 맥락을 관리한다.

## 4. 발표자가 이 조합만 남긴 이유

영상 기준으로 정리하면 이유는 세 가지다.

- 기능이 많아서가 아니라 `작업 흐름에 자연스럽게 붙는다`.
- 사용자가 `도구를 자기 방식대로 제어`할 수 있다.
- 결과물이 억지스럽지 않고 `매끄럽게 연결`된다.

즉, 개별 기능의 화려함보다 `순서가 있는 워크플로우`를 만들어 준다는 점을 더 높게 평가한다.

## 5. 영상 자체의 제작 방식 추정

아래는 영상에서 직접 설명한 내용이 아니라, 스토리보드 프레임을 보고 추정한 연출 방식이다.

### 5-1. 제작 포맷

- 얼굴 캠이나 라이브 코딩 화면 중심이 아니라 `스크립트 기반 설명형 영상`
- 검은 배경 위에 초록색 포인트 컬러를 쓰는 `슬라이드/모션그래픽형 구성`
- 한 화면에 한 메시지씩 던지는 `키네틱 타이포그래피`에 가깝다

### 5-2. 화면 구성 패턴

- 챕터 카드: `1`, `2`, `3` 식으로 단계 번호를 먼저 보여 줌
- 핵심 키워드 카드: `G-Stack`, `Superpowers`, `Git WorkTrees` 같은 용어를 독립 카드로 강조
- 비교 도식: 책상 A/B, 역할 분담, 단계 흐름도 같은 단순 다이어그램 사용
- 하단 자막: 내레이션 문장을 거의 그대로 자막으로 넣어 이해를 보조

### 5-3. 제작 순서 추정

1. 먼저 전체 대본이나 챕터 아웃라인을 작성
2. 챕터별 핵심 문장을 슬라이드 단위로 쪼갬
3. 통일된 템플릿으로 아이콘, 텍스트, 간단한 선형 다이어그램 제작
4. 내레이션 녹음 후 장면 전환 타이밍을 맞춤
5. 자막과 키워드 강조색을 얹어 최종 편집
6. 유튜브 설명란에 타임스탬프를 넣어 챕터 구조를 맞춤

### 5-4. 왜 이런 방식으로 만들었는가

이 주제는 실제 코딩 데모보다 `개념과 순서`를 설득하는 게 중요하다. 그래서 화면 녹화보다 다음 요소가 더 중요했을 가능성이 높다.

- 단계 기억이 쉬운 구조
- 용어 인지성
- 빠른 편집
- 재촬영 부담이 적은 템플릿형 제작

## 6. 실무에 바로 옮길 때의 핵심 포인트

- Claude를 잘 쓰는 핵심은 프롬프트를 길게 넣는 것이 아니라 `순서를 먼저 고정하는 것`이다.
- 이 영상은 구현보다 `사전 정리와 사전 리뷰`를 더 중요하게 둔다.
- 병렬화는 단순 속도 문제가 아니라 `컨텍스트 오염 방지` 문제로 설명된다.
- 계획서 작성과 리뷰를 구현 전에 끝내는 것이 전체 품질을 끌어올리는 핵심 포인트다.

## 7. 요약 결론

이 영상이 제안하는 작업법은 “Claude에게 바로 만들어 달라 하지 말고, 먼저 생각을 구조화한 뒤 설계도와 작업 공간을 분리해서 팀처럼 굴려라”로 정리할 수 있다.

도구 조합으로 쓰면 아래 한 줄이다.

`Office Hours -> 문서화 -> Brainstorming -> Writing-Plans -> Plan Review -> WorkTrees -> Subagents -> Design Review`

이 문서만 따라도 영상의 핵심 워크플로우는 거의 그대로 재현할 수 있다.
