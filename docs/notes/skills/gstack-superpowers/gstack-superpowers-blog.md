# GStack + Superpowers는 어떻게 같이 쓰면 좋을까

Claude Code를 쓰다 보면 금방 느끼는 문제가 하나 있습니다.
처음에는 정말 뭐든 만들어 줄 것 같지만, 조금만 복잡한 작업에 들어가면 방향이 흔들리거나 결과물이 어색해지는 순간이 자주 생깁니다.

문제는 보통 Claude 자체보다, **작업을 시작하는 방식**에 있습니다.
바로 구현부터 들어가면 요구사항이 흐릿한 상태에서 AI가 빈칸을 자기 방식대로 채워 버리기 때문입니다.

`GStack + Superpowers` 조합은 이 지점을 보완해 줍니다.
이 조합의 핵심은 Claude를 더 강한 생성기로 만드는 것이 아니라, **작업 순서를 먼저 잡고 협업 구조를 만드는 것**에 있습니다.

---

## 핵심은 “바로 만들지 않는 것”

이 조합을 한 줄로 요약하면 아래와 같습니다.

`아이디어 정리 -> 문서화 -> 계획 수립 -> 작업 분리 -> 병렬 실행 -> 중간 리뷰`

즉, Claude에게 곧바로 “이거 만들어줘”라고 던지는 방식이 아니라,

- 먼저 생각을 정리하고
- 그 내용을 문서로 남기고
- 실행 계획을 세운 뒤
- 작업 공간을 분리해서
- 병렬로 구현하고
- 중간중간 결과물을 다듬는 흐름

으로 가져가는 방식입니다.

이 순서가 잡히면 결과물이 훨씬 덜 흔들리고, 작업이 “한 번에 잘 되는 느낌”에 가까워집니다.

---

## GStack는 어디에 쓰는가

GStack는 시작과 점검 단계에서 힘을 발휘합니다.

### 1. Office Hours

`Office Hours`는 작업 시작 전에 Claude와 짧은 인터뷰를 하는 단계입니다.

예를 들면 이런 질문을 먼저 정리하게 됩니다.

- 왜 이 기능이 필요한가
- 누가 어떤 상황에서 쓰는가
- 지금 불편한 점은 무엇인가
- 비슷한 방식으로 해봤을 때 뭐가 아쉬웠는가

이 과정을 거치면, 막연했던 아이디어가 꽤 선명해집니다.
바로 구현에 들어갔을 때 자주 생기는 “이건 아닌데?”를 초반에 많이 줄일 수 있습니다.

### 2. Design Review

`Design Review`는 구현 중간이나 결과물을 본 뒤에 쓰는 점검 단계입니다.

특히 AI로 빠르게 만든 결과물은 얼핏 그럴듯해 보여도,

- 과한 스타일
- 어색한 UI 디테일
- 전형적인 AI 느낌

이 남아 있는 경우가 많습니다.

이럴 때 Design Review를 쓰면 결과물을 한 번 더 정리하는 역할을 합니다.
쉽게 말하면, “만드는 단계”가 아니라 “다듬는 단계”에 가깝습니다.

---

## Superpowers는 어디에 쓰는가

Superpowers는 생각을 실행 가능한 흐름으로 바꾸고, 실제 구현을 분리해서 굴리는 데 강합니다.

### 1. Brainstorming

Office Hours나 초안 문서가 나온 다음에 `Brainstorming`을 붙이면 좋습니다.

이 단계에서는,

- 빠진 요구사항은 없는지
- 방향이 맞는지
- UX/UI 측면에서 놓친 부분은 없는지

를 한 번 더 점검할 수 있습니다.

처음 정리한 아이디어를 다시 보는 단계라고 생각하면 됩니다.
실제로는 이 한 번의 점검이 초반 설계 구멍을 꽤 많이 줄여 줍니다.

### 2. Writing-Plans

`Writing-Plans`는 구현 전에 계획서를 만드는 단계입니다.

이게 중요한 이유는 단순합니다.
큰 작업은 중간에 쉽게 흔들리기 때문입니다.

계획서가 있으면,

- 지금 어디까지 왔는지 알 수 있고
- 다음에 뭘 해야 하는지 명확하고
- 막혔을 때 어디로 돌아가야 하는지도 보입니다

결국 구현을 잘하기 위한 핵심은 더 많은 프롬프트가 아니라, **작업 기준점이 있는가**에 가깝습니다.

### 3. Git WorkTrees

작업이 커지면 한 공간에서 모든 걸 같이 처리하는 방식이 금방 꼬입니다.

`Git WorkTrees`는 작업 공간을 기능별로 나누는 데 쓰입니다.

예를 들어,

- 기능 A
- UI 수정
- 문서 보완
- 버그 수정

같은 작업이 한 공간에 다 섞여 있으면 컨텍스트가 오염되기 쉽습니다.
반대로 작업 공간을 나눠 두면 각 작업이 서로 영향을 덜 주고, 훨씬 안정적으로 진행됩니다.

### 4. Subagent Driven Development

작업 공간을 나눈 다음에는 `Subagent Driven Development`로 병렬 실행을 붙일 수 있습니다.

이 기능은 큰 작업을 한 번에 한 컨텍스트 안에서 처리하는 대신,
작은 단위로 나눠 각각 집중하게 만드는 방식입니다.

쉽게 말하면 한 사람이 다 하는 게 아니라,

- 기획처럼 보는 시점
- 구현에 집중하는 시점
- 리뷰에 집중하는 시점

을 분리해서 운영하는 느낌에 가깝습니다.

큰 작업일수록 이 방식의 차이가 커집니다.

### 5. 자동 기록

Superpowers의 장점 중 하나는 작업 기록을 자연스럽게 남긴다는 점입니다.

작업을 하다 보면 “이게 언제 바뀌었지?”가 자주 생기는데,
기록이 잘 남아 있으면 흐름을 다시 잡기가 쉬워집니다.

이건 겉보기에 작아 보여도 실제 작업 효율에는 꽤 큰 영향을 줍니다.

---

## 실제로는 어떤 순서로 쓰면 되나

실무에서는 아래 순서로 가져가면 가장 자연스럽습니다.

1. `Office Hours`로 시작한다.
2. 대화 내용을 바탕으로 작업 문서를 정리한다.
3. `Brainstorming`으로 빠진 내용을 다시 점검한다.
4. `Writing-Plans`로 단계별 계획서를 만든다.
5. 계획서를 리뷰해서 한 번 더 검증한다.
6. `Git WorkTrees`로 작업 공간을 나눈다.
7. `Subagent Driven Development`로 구현을 병렬 진행한다.
8. 중간중간 `Design Review`로 결과물을 다듬는다.
9. 자동 기록을 활용해 변경 이력을 관리한다.

이 흐름의 좋은 점은, 처음부터 끝까지 작업의 선이 끊기지 않는다는 데 있습니다.

---

## 이 방식이 특히 잘 맞는 경우

다음 같은 작업에서는 이 조합의 장점이 분명하게 드러납니다.

- 새 기능 기획부터 구현까지 한 번에 진행해야 할 때
- 요구사항이 아직 완전히 정리되지 않았을 때
- UI, UX, 문서, 구현이 함께 얽혀 있을 때
- 동시에 여러 단위를 병렬로 진행해야 할 때
- 작업할수록 방향이 자주 흔들리던 팀이나 프로젝트일 때

반대로 아주 작은 수정 하나만 빠르게 끝내는 작업이라면 이 전체 흐름을 다 쓸 필요는 없습니다.
하지만 기능 단위 이상의 작업에서는 꽤 안정적인 방식입니다.

---

## 왜 이 조합이 좋은가

이 조합의 장점은 “기능이 많다”가 아닙니다.
좋은 이유는 아래에 더 가깝습니다.

- 시작 전에 생각을 정리하게 해 준다
- 구현 전에 계획을 세우게 해 준다
- 작업을 섞지 않고 분리하게 해 준다
- 큰 작업을 병렬로 굴릴 수 있게 해 준다
- 결과물을 중간에 한 번 더 다듬게 해 준다

결국 이건 툴 조합이라기보다, **AI와 일하는 운영 방식**에 가깝습니다.

---

## 한 줄로 정리하면

`GStack + Superpowers`는 Claude Code를 더 똑똑하게 만드는 조합이라기보다,
**작업을 잘 시작하고, 잘 나누고, 잘 끝내게 만드는 조합**입니다.

실무용으로 가장 짧게 정리하면 이 문장으로 충분합니다.

`바로 만들지 말고, 먼저 정리하고, 계획을 세우고, 나눠서 진행한다.`
