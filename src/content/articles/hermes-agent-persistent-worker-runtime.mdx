Hermes Agent를 한 문장으로 줄이면 “터미널에서 쓰는 챗봇”이 아니라, 모델을 두뇌로 쓰고 터미널, 파일, 브라우저, 메모리, 스킬, 크론, 메시징 채널을 손발로 쓰는 개인 AI 워커 런타임이다.

이번 글은 사용자가 제공한 한국어 심층 리포트를 그대로 옮긴 글이 아니다. 2026년 6월 29일 Asia/Seoul 기준으로 `NousResearch/hermes-agent`를 로컬에 직접 클론하고, README, 공식 문서, `run_agent.py`, `model_tools.py`, `tools/registry.py`, `toolsets.py`, `hermes_state.py`, `tools/session_search_tool.py`, `tools/skill_manager_tool.py`, `cron/scheduler.py`를 다시 확인했다. 로컬 확인 기준 커밋은 `d0d2cf1c2f7e821e6d06a7a0e838ad66c6e17fd5`다.

검증 결과 원문 리포트의 큰 방향은 맞았다. 다만 글로 공개할 때는 더 선명하게 말해야 한다. Hermes의 본질은 “self-improving AI assistant”라는 마케팅 문장보다 “권한을 가진 상주형 자동화 작업자”에 가깝다. 이 차이를 이해하지 못하면 장점도 놓치고, 보안 리스크도 과소평가하게 된다.

요약하면 이렇게 본다.

| 질문 | 답 |
| --- | --- |
| 이게 뭔가? | CLI, 메시징 게이트웨이, 장기 기억, 스킬, 도구 실행, 크론을 한 런타임에 묶은 AI 에이전트 시스템이다. |
| 핵심 가치는? | 같은 에이전트를 터미널, Telegram, Slack, Email, Cron 같은 표면에서 계속 쓰며 기억과 절차를 축적하는 데 있다. |
| 코드 품질 신호는? | tool registry, SQLite/FTS5 세션 저장소, cron fail-closed, gateway authorization, transient probe hardening처럼 실제 운영 이슈를 반영한 흔적이 많다. |
| 가장 큰 위험은? | terminal/file/browser/messaging/cron 권한이 넓다. 편한 비서가 아니라 권한 있는 worker로 배치해야 한다. |
| 추천 시작점은? | local backend와 YOLO가 아니라 Docker 또는 SSH backend, manual approval, memory write approval, 좁은 toolset부터 시작하는 것이다. |

## 먼저 알아야 할 개념

주니어 개발자 입장에서 Hermes를 이해하려면 “모델”, “도구”, “런타임”을 나눠 생각하면 쉽다.

모델은 답을 생성하는 두뇌다. OpenAI, Anthropic, OpenRouter, Nous Portal 같은 provider가 여기에 들어간다. 그런데 모델만 있으면 파일을 고치거나 명령을 실행할 수 없다. 그래서 Hermes는 모델에게 여러 도구를 붙인다. 웹 검색, 파일 읽기와 패치, 터미널 실행, 브라우저 조작, 이미지 분석, 스킬 조회, 메모리 저장, 세션 검색, 크론 등록 같은 것들이다.

런타임은 그 도구들을 언제, 어떤 권한으로, 어떤 대화 기록과 함께 실행할지 관리하는 층이다. Hermes의 중심에는 `run_agent.py`의 `AIAgent`가 있고, CLI, Gateway, Cron, ACP, Batch Runner 같은 여러 진입점이 결국 이 agent loop로 수렴한다. 공식 architecture 문서도 이 구조를 명확히 그린다.

여기서 “self-improving”이라는 표현을 오해하면 안 된다. Hermes가 모델 가중치를 스스로 학습한다는 뜻은 아니다. 더 정확히는 다음 실행을 더 잘하기 위해 기억과 절차를 축적한다는 뜻이다.

| 축 | 저장되는 것 | 왜 중요한가 |
| --- | --- | --- |
| Memory | 환경, 사용자 선호, 프로젝트 규칙 같은 짧은 사실 | 매 세션 시작부터 기본 맥락으로 들어간다. |
| Session Search | 과거 대화 전체 | 기억에 없는 오래된 논의도 SQLite/FTS5로 다시 찾는다. |
| Skills | 반복 업무 절차 | “어떻게 할지”를 `SKILL.md`로 저장해 재사용한다. |
| Cron | 주기 작업 | fresh agent session을 일정에 따라 실행하고 결과를 메시징 채널로 보낸다. |

그래서 Hermes는 일반 챗봇보다 더 지속적이다. 동시에 더 위험하다. 기억하고, 실행하고, 예약하고, 외부 메시지를 받기 때문이다.

## 실제 구조

공식 architecture 문서는 Hermes의 진입점을 CLI, Gateway, ACP, Batch Runner, API Server, Python Library로 나눈다. 이들은 모두 `AIAgent`로 들어가고, 그 아래에서 prompt assembly, provider resolution, tool dispatch, session storage, terminal/browser/web/MCP/file/vision backend가 움직인다.

소스 구조를 실제로 보면 중심 파일은 다음처럼 나뉜다.

| 영역 | 파일 | 역할 |
| --- | --- | --- |
| Agent loop | `run_agent.py` | system prompt 조립, provider/API mode 선택, tool call loop, retry, fallback, compression |
| Tool orchestration | `model_tools.py` | tool schema 수집, registry discovery, sync/async bridge, dispatch API |
| Tool registry | `tools/registry.py` | 각 tool file의 self-registration, availability check, schema snapshot |
| Toolset policy | `toolsets.py` | CLI, gateway, webhook, cron 등 표면별 tool 노출 묶음 |
| Session DB | `hermes_state.py` | SQLite state store, WAL fallback, FTS5 search, session lineage |
| Recall | `tools/session_search_tool.py` | 과거 대화 discovery, scroll, browse |
| Skills | `tools/skill_manager_tool.py` | agent-managed skill create/edit/patch/delete |
| Cron | `cron/scheduler.py` | scheduled agent session 실행, delivery, fail-closed 정책 |

이 구조에서 중요한 것은 toolset이다. toolset은 단순한 UI 분류가 아니라 모델에게 어떤 도구 schema를 보여줄지 결정하는 1차 권한 경계다. `toolsets.py`의 core tools에는 web, terminal, file, browser, vision, image generation, skills, todo, memory, session_search, clarify, execute_code, delegation, cronjob, Home Assistant, kanban, computer use까지 들어간다.

반면 webhook toolset은 기본적으로 `web_search`, `web_extract`, `vision_analyze`, `clarify` 정도로 제한된다. 코드 주석도 공개 PR 제목이나 댓글 같은 untrusted third-party content가 들어올 수 있기 때문에 local file/system execution을 피하려는 설계라고 설명한다. 이건 좋은 신호다. Hermes가 “모든 곳에 모든 권한”을 주는 구조가 아니라, 표면별 위험을 적어도 인식하고 있다는 뜻이다.

패키지 메타데이터도 운영 성향이 강하다. `pyproject.toml`은 `hermes-agent` `0.17.0`, Python `>=3.11,<3.14`를 선언하고, core dependency를 exact pin 중심으로 관리한다. 주석에는 supply-chain 공격과 transitive dependency drift를 줄이기 위한 판단까지 적혀 있다. 개인용 데모 프로젝트라기보다 실제 설치 환경에서 깨지는 문제를 꽤 의식한 코드다.

## 핵심 기능

첫 번째 핵심은 여러 표면에서 같은 agent를 쓰는 것이다. README는 CLI뿐 아니라 Telegram, Discord, Slack, WhatsApp, Signal, Email 같은 메시징 게이트웨이를 전면에 둔다. Gateway message flow는 platform event를 adapter가 받고, authorization을 확인하고, session key를 결정한 뒤, session history가 붙은 `AIAgent`를 만들어 응답을 다시 platform adapter로 보내는 형태다.

두 번째 핵심은 장기 기억이다. 공식 memory 문서는 `MEMORY.md`와 `USER.md` 두 파일을 설명한다. `MEMORY.md`는 환경과 프로젝트 규칙 같은 agent-side notes, `USER.md`는 사용자 선호와 기대치다. 둘 다 `~/.hermes/memories/` 아래 저장되고, 세션 시작 시 frozen snapshot으로 system prompt에 들어간다. 세션 중 memory write는 disk에는 저장되지만 active prompt에 즉시 재주입되지는 않는다. prompt cache를 지키기 위한 의도적 설계다.

세 번째 핵심은 session search다. `tools/session_search_tool.py`는 FTS5 기반으로 과거 대화를 찾고, LLM 요약이 아니라 DB에 저장된 실제 메시지를 반환한다. discovery, scroll, browse 세 호출 형태를 args 모양으로 추론한다. 흥미로운 점은 cron session을 아예 제외하지 않고 검색 가능하게 두되 interactive session보다 rank를 낮춘다는 것이다. 반복 cron output이 검색 결과를 덮어 사용자의 실제 대화가 안 보이는 “recall blindness”를 막으려는 설계다.

네 번째 핵심은 skills다. 공식 문서와 `skill_manager_tool.py`는 skills를 procedural memory로 설명한다. 일반 memory가 “무엇을 기억할지”라면 skill은 “어떻게 할지”를 저장한다. `/learn`은 로컬 문서, 온라인 문서, 방금 수행한 workflow, 붙여넣은 절차를 `SKILL.md`로 바꾸는 경로다. Agent가 skill을 create, edit, patch, delete할 수 있으므로 강력하지만 drift 위험도 있다.

다섯 번째 핵심은 cron이다. Hermes cron은 단순히 shell command를 예약 실행하는 기능이 아니다. 공식 cron 문서와 `cron/scheduler.py`를 보면 due job마다 fresh `AIAgent` session을 만들고, skill을 주입하고, prompt를 실행하고, 결과를 origin chat, local file, Telegram, Discord, Slack, Email 등으로 보낸다. no-agent mode도 있어서 단순 script stdout delivery는 LLM 없이 처리할 수 있다.

실제 사용자 이슈도 이 사용 방식을 뒷받침한다. GitHub 이슈에는 Telegram daily briefing, Telegram rich message table 문제, Slack Block Kit 요구, Email gateway subject isolation, cron `deliver=origin` target loss, sandbox file transfer, Telegram photo caching, provider switch poisoning, Windows/WSL remote backend, macOS launchd service, QQBot adapter 같은 사례가 있었다. 일부는 closed 상태지만, Hermes가 개인 CLI 도구를 넘어 메시징과 자동화 표면에서 실제로 쓰이고 있다는 근거로는 충분하다.

## 좋은 점

가장 큰 장점은 “지속성”이다. 일반적인 웹 챗봇은 대화 하나가 끝나면 맥락이 약해진다. Hermes는 memory, session search, skills, cron으로 이전 대화와 반복 절차를 다시 사용할 수 있게 만든다. 개인 비서, 리서치 워커, 운영 점검 봇, 반복 개발 작업 도우미에 잘 맞는 방향이다.

두 번째 장점은 tool runtime이 생각보다 실전적이라는 점이다. `model_tools.py`는 async tool handler를 sync 경로에서 안전하게 돌리기 위해 persistent event loop와 per-worker loop를 쓴다. 예전처럼 `asyncio.run()`을 매번 쓰면 cached httpx 또는 AsyncOpenAI client가 죽은 event loop에 묶여 “Event loop is closed” 오류를 만들 수 있기 때문이다.

세 번째 장점은 lazy MCP discovery 제거다. `model_tools.py` 주석은 MCP discovery를 import side effect에서 제거한 이유를 설명한다. gateway가 asyncio event loop 안에서 처음 메시지를 받을 때 MCP discovery가 120초까지 blocking되면 Discord/Telegram heartbeat가 얼 수 있기 때문이다. 이건 장기 실행 gateway에서 실제로 중요한 운영 품질이다.

네 번째 장점은 tool availability hardening이다. `tools/registry.py`는 Docker daemon, Modal SDK, Playwright binary 같은 외부 상태 probe를 매번 새로 하지 않고 TTL cache를 둔다. 더 중요한 점은 최근 성공 직후의 일시 실패를 flake로 보고 last-good true를 짧게 유지한다는 것이다. 순간적인 Docker probe timeout 때문에 subagent에서 file/terminal toolset이 갑자기 사라지는 문제를 줄이는 장치다.

다섯 번째 장점은 session DB hardening이다. `hermes_state.py`는 SQLite WAL mode를 기본으로 쓰되 NFS, SMB, 일부 FUSE처럼 WAL locking이 깨지는 환경에서는 DELETE journal mode로 fallback한다. macOS에서는 checkpoint full fsync barrier도 적용한다. 단순한 “SQLite에 저장합니다”보다 훨씬 운영 문제를 의식한 설계다.

여섯 번째 장점은 cron 비용/권한 guard다. 공식 cron 문서는 provider/model을 명시하지 않은 job이 생성 시점의 provider/model snapshot을 잡고, 나중에 global default가 바뀌면 fail closed한다고 설명한다. `cron/scheduler.py`도 cron-spawned agent에서 `cronjob`, `messaging`, `clarify` toolset을 항상 비활성화한다. unattended job이 cron을 무한 생성하거나, interactive 사용자 응답을 기다리며 멈추는 일을 막기 위한 판단이다.

## 조심해야 할 점

첫 번째 리스크는 권한이 넓다는 점이다. Hermes는 terminal, file, browser, messaging, cron을 다룬다. 잘못 세팅하면 agent가 로컬 파일을 읽고 쓰고, 명령을 실행하고, 브라우저를 조작하고, 메시징 플랫폼에서 외부 요청을 받아 처리하고, 예약 작업으로 무인 실행할 수 있다. “AI assistant 설치”가 아니라 “권한 있는 worker 배치”로 봐야 한다.

두 번째 리스크는 approval 설정이다. 보안 문서는 `approvals.mode`가 `manual`, `smart`, `off`를 지원한다고 설명한다. `off`는 모든 approval check를 끄는 모드이고, YOLO도 위험 명령 승인 prompt를 우회한다. hardline blocklist가 항상 남아 있기는 하지만, 이것은 최후의 바닥일 뿐이다. 시작 설정으로는 `manual`과 `cron_mode: deny`가 맞다.

세 번째 리스크는 container backend에 대한 오해다. 보안 문서는 Docker, Singularity, Modal, Daytona backend에서는 container가 security boundary라 dangerous command check가 skip된다고 설명한다. 동시에 tools 문서는 Docker backend가 fresh container per command가 아니라 process lifetime 동안 유지되는 persistent sandbox라고 설명한다. 패키지 설치, cwd, 파일, 환경 조정이 다음 tool call과 subagent에 남는다. 편하지만 오염도 남는다.

네 번째 리스크는 memory와 skills drift다. Memory는 세션 시작 prompt에 들어가므로 틀린 가정이 저장되면 다음 세션을 계속 오염시킨다. Skills는 더 강하다. 잘못된 절차가 `SKILL.md`로 저장되면 agent가 같은 실수를 반복할 수 있다. 그래서 `memory.write_approval: true`, skill folder Git 관리, 민감 skill 보호, `/learn` 결과 리뷰가 중요하다.

다섯 번째 리스크는 messaging UX다. 실제 이슈를 보면 Telegram Markdown/table escaping, Telegram image caching timeout, Email notification volume, Slack Block Kit 부족, origin delivery target loss 같은 문제가 나온다. 기능 자체보다 platform별 delivery reliability와 session isolation이 품질을 좌우한다.

여섯 번째 리스크는 toolset을 보안 경계로 과신하는 것이다. Toolset은 모델에게 노출되는 schema를 줄이는 경계다. OS 권한, 네트워크 접근, 파일 시스템 권한을 완전히 격리하는 것은 아니다. 실제 보안 경계는 Docker/SSH/Modal 같은 backend, gateway allowlist, approval mode, env passthrough 최소화, workdir 제한, API key scope 제한과 함께 만들어야 한다.

## 언제 쓰면 좋은가

Hermes가 잘 맞는 사용자는 여러 프로젝트와 반복 업무를 장기간 다루는 사람이다. 매번 같은 환경 설명을 해야 하고, 이전 대화에서 결정한 내용을 다시 찾고, Telegram이나 Slack에서 작업을 던지고, 매일/매시간 리서치나 운영 점검을 자동화하고 싶다면 Hermes의 구조가 잘 맞는다.

개인 파워유저에게는 특히 매력적이다. 예를 들어 아침에 Telegram으로 시장/뉴스/프로젝트 브리핑을 받고, 낮에는 CLI에서 repo 테스트 실패를 분석하고, 저녁에는 Slack에서 긴 이슈 요약을 시키는 식으로 같은 런타임을 여러 표면에서 쓸 수 있다. Memory와 session search가 쌓이면 “지난번 우리 프로젝트에서 왜 이렇게 했지?” 같은 질문에도 답할 가능성이 커진다.

개발자에게는 agent harness 연구 대상으로도 가치가 있다. Provider resolution, tool registry, progressive skill disclosure, persistent session store, gateway authorization, cron fail-closed 같은 구현을 실제 코드로 볼 수 있다. AI agent runtime을 만들고 싶은 사람에게는 좋은 참고 자료다.

반대로 단순한 질문 답변, deterministic한 SaaS integration, 엄격한 감사 환경, 공개 workspace에 바로 agent를 열어야 하는 팀에는 신중해야 한다. Zapier나 n8n 같은 workflow tool은 예측 가능성이 강하고, Hermes는 LLM 판단과 권한 실행이 섞인다. 열린 문제를 다루는 데는 강하지만, 감사 가능한 deterministic workflow에는 별도 설계가 필요하다.

도입한다면 단계적으로 가는 편이 좋다.

| 단계 | 목표 | 권장 설정 |
| --- | --- | --- |
| 0 | 로컬 실험 | messaging gateway 끄기, `terminal.backend=docker` 또는 `ssh`, approval manual |
| 1 | 단발 CLI 작업 | `web,file,terminal,todo,skills`처럼 좁은 toolset |
| 2 | Memory 안정화 | `memory.write_approval=true`, `/memory pending` 리뷰 |
| 3 | Skills 축적 | 반복 workflow를 3번 이상 수행한 뒤 `/learn`, 결과 Git diff 리뷰 |
| 4 | Gateway 연결 | allowlist 필수, trusted DM부터, public/group은 read-only toolset |
| 5 | Cron 자동화 | read-only report부터, provider/model pinning, delivery fallback |

## 주니어 개발자가 배울 점

첫째, 에이전트의 품질은 모델 성능만으로 결정되지 않는다. 어떤 도구를 보여줄지, 어떤 상태를 기억할지, 실패했을 때 어떻게 fallback할지, 결과를 어디에 저장할지 같은 런타임 설계가 실제 품질을 만든다.

둘째, “기억”은 공짜가 아니다. Memory는 매번 prompt에 들어가므로 작고 정확해야 한다. Session search는 필요할 때만 찾는 recall layer다. Skills는 반복 절차를 저장하는 executable documentation에 가깝다. 이 셋을 구분하면 많은 AI 도구의 설계를 더 잘 읽을 수 있다.

셋째, 자동화는 권한 설계와 함께 봐야 한다. Cron은 편하지만 unattended execution이다. Messaging gateway는 편하지만 외부 입력이 agent 권한으로 들어오는 길이다. Terminal backend는 편하지만 host 손상 가능성과 secret 노출 가능성을 함께 가져온다.

넷째, 좋은 코드베이스는 운영 이슈의 흔적을 남긴다. Hermes 코드에서 인상적인 부분은 화려한 기능 이름보다 event loop 재사용, WAL fallback, last-good availability cache, provider/model drift fail-closed, cron prompt injection scan 같은 작은 방어선이다. 이런 장치들은 실제 사용자가 겪은 실패에서 나온다.

다섯째, “작게 시작하는 설정”이 실력이다. 처음부터 full tools, local backend, YOLO, gateway allow-all, auto memory write를 켜는 것은 빠르지만 위험하다. 좁은 toolset, sandboxed backend, approval, allowlist, staged memory review로 시작해 필요한 권한만 넓히는 편이 프로답다.

## 내 결론

Hermes Agent는 매우 흥미로운 프로젝트다. 단순 코딩 에이전트가 아니라 개인 AI 운영체제에 가까운 방향을 본다. CLI에서만 일하는 도구가 아니라 gateway와 cron을 통해 계속 살아 있고, memory와 skills를 통해 다음 작업에 영향을 주며, 여러 provider와 tool backend를 하나의 agent loop로 묶는다.

좋은 점은 분명하다. 지속성, 멀티채널, 자동화, 스킬화, 소스 수준의 운영 hardening이 있다. 특히 `AIAgent`로 수렴하는 구조, registry 기반 tool system, SQLite/FTS5 session recall, cron의 fresh agent session 모델은 agent runtime을 공부하는 사람에게 볼 가치가 크다.

하지만 이 프로젝트를 “편한 AI 비서”로만 소개하면 위험하다. Hermes는 terminal/file/browser/messaging/cron 권한을 가진 worker다. 설치 직후부터 강하게 쓰기보다, Docker 또는 SSH backend, manual approval, memory write approval, gateway allowlist, 좁은 toolset으로 시작해야 한다. Cron은 처음엔 no-agent 또는 read-only report부터, skill은 Git으로 관리하면서 키우는 편이 맞다.

내가 추천하는 한 줄은 이렇다. Hermes는 믿고 맡기는 제품이라기보다, 권한을 설계하며 키우는 개인 AI 워커 런타임이다. 그 관점으로 접근하면 강력하고, 그 관점을 놓치면 위험하다.
