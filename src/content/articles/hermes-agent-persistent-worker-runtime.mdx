Hermes Agent를 처음 보면 “터미널에서 쓰는 AI 비서” 정도로 보이기 쉽다. 그런데 저장소를 직접 열어보면 결이 조금 다르다. 이 프로젝트는 채팅 인터페이스 하나를 잘 만든 도구라기보다, 모델에게 파일, 터미널, 브라우저, 메시징, 기억, 스킬, 크론을 붙여 오래 굴리는 개인용 AI 워커 런타임에 가깝다.

이번 글은 사용자가 제공한 한국어 심층 리포트를 그대로 옮긴 것이 아니다. 2026년 6월 29일 Asia/Seoul 기준으로 `NousResearch/hermes-agent`를 로컬에 클론했고, README와 공식 문서, 그리고 `run_agent.py`, `model_tools.py`, `tools/registry.py`, `toolsets.py`, `hermes_state.py`, `tools/session_search_tool.py`, `tools/skill_manager_tool.py`, `cron/scheduler.py`를 직접 확인했다. 로컬에서 본 커밋은 `d0d2cf1c2f7e821e6d06a7a0e838ad66c6e17fd5`다.

내 판단은 이렇다. 원문 리포트의 큰 방향은 맞다. 다만 공개 글로 정리할 때는 “self-improving assistant”라는 말보다 “권한을 가진 상주형 자동화 작업자”라는 표현이 훨씬 정확하다. Hermes는 편한 도구이기도 하지만, 동시에 권한 설계를 잘못하면 꽤 위험한 도구다. 이 글은 그 둘을 같이 보려는 리뷰다.

처음 읽는다면 아래 정도만 잡고 들어가도 충분하다.

| 관점 | 짧은 판단 |
| --- | --- |
| 정체성 | CLI 챗봇이 아니라 여러 도구와 채널을 묶은 agent runtime이다. |
| 매력 | 같은 agent를 터미널, Telegram, Slack, Email, Cron에서 계속 쓰며 기억과 절차를 쌓을 수 있다. |
| 코드 신호 | tool registry, SQLite/FTS5 recall, cron fail-closed, gateway authorization처럼 실제 운영 문제를 만져본 흔적이 있다. |
| 리스크 | terminal/file/browser/messaging/cron 권한이 넓다. 비서가 아니라 worker로 배치해야 한다. |
| 첫 도입 | local backend와 YOLO부터 켜기보다 Docker 또는 SSH backend, manual approval, memory write approval, 좁은 toolset부터 시작하는 편이 낫다. |

## 먼저 알아야 할 개념

주니어 개발자라면 Hermes를 “모델”, “도구”, “런타임”으로 나눠 보면 이해가 빠르다.

모델은 답을 만드는 두뇌다. OpenAI, Anthropic, OpenRouter, Nous Portal 같은 provider가 여기에 해당한다. 하지만 모델만으로는 파일을 고치거나 명령을 실행할 수 없다. 그래서 Hermes는 모델에게 여러 도구를 붙인다. 웹 검색, 파일 읽기와 패치, 터미널 실행, 브라우저 조작, 이미지 분석, 스킬 조회, 메모리 저장, 세션 검색, 크론 등록 같은 것들이다.

런타임은 그 도구들을 어떤 맥락과 권한으로 실행할지 관리하는 층이다. Hermes의 중심에는 `run_agent.py`의 `AIAgent`가 있다. CLI, Gateway, Cron, ACP, Batch Runner 같은 진입점이 서로 달라 보여도 결국 이 agent loop로 들어간다. 공식 architecture 문서도 이 수렴 구조를 전면에 둔다.

여기서 “self-improving”이라는 표현은 조심해서 읽어야 한다. 모델 가중치를 스스로 학습한다는 뜻이 아니다. Hermes가 말하는 개선은 대체로 다음 실행을 더 잘하기 위해 기억과 절차를 축적하는 쪽에 가깝다.

| 축 | 저장되는 것 | 실제 의미 |
| --- | --- | --- |
| Memory | 환경, 사용자 선호, 프로젝트 규칙 같은 짧은 사실 | 매 세션 시작부터 기본 맥락으로 들어간다. |
| Session Search | 과거 대화 전체 | 기억에 없던 논의도 SQLite/FTS5로 다시 찾는다. |
| Skills | 반복 업무 절차 | “어떻게 처리할지”를 `SKILL.md`로 남겨 재사용한다. |
| Cron | 주기 작업 | 일정에 따라 새 agent session을 만들고 결과를 메시징 채널로 보낸다. |

즉 Hermes는 일반 챗봇보다 훨씬 지속적이다. 기억하고, 실행하고, 예약하고, 외부 메시지를 받는다. 이 지속성이 장점이지만, 같은 이유로 배치할 때 훨씬 신중해야 한다.

## 실제 구조

코드를 읽을 때 제일 먼저 확인한 파일은 `run_agent.py`였다. 여기에는 system prompt 조립, provider/API mode 선택, tool call loop, retry, fallback, context compression 같은 agent loop의 중심 로직이 들어 있다. Hermes의 여러 기능이 복잡해 보이지만, 결국 “프롬프트를 만들고, 모델을 부르고, tool call을 처리하고, 다음 메시지로 이어가는” 루프 위에 얹힌다.

그 다음으로 중요한 파일은 `model_tools.py`와 `tools/registry.py`다. `model_tools.py`는 모델에게 노출할 tool schema를 모으고 실제 tool call을 dispatch한다. `tools/registry.py`는 각 tool file이 스스로 등록되는 구조를 관리한다. 도구가 많아질수록 “어떤 도구가 언제 사용 가능한가”가 품질과 안정성을 좌우하는데, Hermes는 이 부분을 꽤 명시적으로 다룬다.

권한 관점에서는 `toolsets.py`가 중요하다. toolset은 단순한 UI 분류가 아니다. 모델에게 어떤 tool schema를 보여줄지 정하는 1차 권한 경계다. core toolset에는 web, terminal, file, browser, vision, image generation, skills, todo, memory, session search, execute code, delegation, cronjob, Home Assistant, kanban, computer use까지 들어간다. 반면 webhook toolset은 기본적으로 `web_search`, `web_extract`, `vision_analyze`, `clarify` 정도로 제한된다.

이 차이가 꽤 중요하다. webhook에는 GitHub PR 제목이나 댓글처럼 신뢰할 수 없는 외부 입력이 들어올 수 있다. Hermes 코드 주석도 이 표면에서 local file/system execution을 피하려는 의도를 설명한다. 모든 표면에 모든 권한을 주는 구조가 아니라, 입력 표면별 위험을 적어도 코드 레벨에서 인식하고 있다는 뜻이다.

나머지 핵심 파일은 역할이 비교적 선명하다.

| 영역 | 파일 | 볼 만한 지점 |
| --- | --- | --- |
| Agent loop | `run_agent.py` | provider 선택, tool loop, retry, compression |
| Tool orchestration | `model_tools.py` | schema 수집, async/sync bridge, dispatch |
| Tool registry | `tools/registry.py` | self-registration, availability check, schema snapshot |
| Toolset policy | `toolsets.py` | 표면별 tool 노출 정책 |
| Session DB | `hermes_state.py` | SQLite state store, WAL fallback, FTS5 search |
| Recall | `tools/session_search_tool.py` | 과거 대화 discovery, scroll, browse |
| Skills | `tools/skill_manager_tool.py` | agent-managed skill create/edit/patch/delete |
| Cron | `cron/scheduler.py` | scheduled agent session, delivery, fail-closed 정책 |

`pyproject.toml`도 그냥 지나칠 파일은 아니었다. 패키지는 `hermes-agent` `0.17.0`, Python `>=3.11,<3.14`를 선언한다. core dependency를 exact pin 중심으로 관리하고, 주석에는 supply-chain 공격과 transitive dependency drift를 줄이려는 판단이 적혀 있다. 개인 실험용 스크립트라기보다, 설치해서 오래 돌릴 때 깨지는 문제를 신경 쓴 프로젝트라는 인상을 준다.

## 핵심 기능

Hermes의 첫 번째 특징은 같은 agent를 여러 표면에서 쓴다는 점이다. README는 CLI뿐 아니라 Telegram, Discord, Slack, WhatsApp, Signal, Email 같은 메시징 게이트웨이를 크게 다룬다. Gateway flow는 platform event를 adapter가 받고, authorization을 확인하고, session key를 정한 뒤, session history가 붙은 `AIAgent`를 만들어 응답을 다시 platform adapter로 보내는 형태다.

이 구조가 왜 중요할까. 단순한 CLI agent라면 터미널을 닫는 순간 사용 맥락도 같이 끊긴다. Hermes는 Telegram에서 던진 작업, CLI에서 이어간 작업, cron이 만든 결과를 같은 runtime의 기억과 세션 저장소 위에서 다룰 수 있게 하려 한다.

두 번째 특징은 장기 기억이다. 공식 memory 문서는 `MEMORY.md`와 `USER.md`를 설명한다. `MEMORY.md`는 환경과 프로젝트 규칙 같은 agent-side notes이고, `USER.md`는 사용자 선호와 기대치다. 둘 다 `~/.hermes/memories/` 아래 저장되고, 세션 시작 시 frozen snapshot으로 system prompt에 들어간다. 세션 중 memory write가 disk에는 저장되지만 active prompt에 즉시 다시 들어가지는 않는다는 점도 흥미롭다. prompt cache를 지키기 위한 의도적인 설계다.

세 번째 특징은 session search다. `tools/session_search_tool.py`를 보면 과거 대화를 FTS5 기반으로 찾고, LLM이 새로 요약한 내용이 아니라 DB에 저장된 실제 메시지를 반환한다. discovery, scroll, browse 형태를 args 모양으로 추론하는 구현도 있다. 작은 디테일이지만 cron session을 완전히 제외하지 않고 interactive session보다 rank만 낮춘 것도 좋았다. 반복 cron output이 검색 결과를 덮어 사용자의 실제 대화가 안 보이는 문제를 줄이려는 판단으로 읽힌다.

네 번째 특징은 skills다. 공식 문서와 `skill_manager_tool.py`는 skills를 procedural memory로 설명한다. 일반 memory가 “무엇을 기억할지”라면, skill은 “어떻게 할지”를 저장한다. `/learn`은 로컬 문서, 온라인 문서, 방금 수행한 workflow, 붙여넣은 절차를 `SKILL.md`로 바꾸는 경로다. Agent가 skill을 create, edit, patch, delete할 수 있으므로 강력하지만, 그만큼 잘못된 절차가 굳어질 위험도 있다.

다섯 번째 특징은 cron이다. Hermes cron은 shell command를 예약 실행하는 수준이 아니다. 공식 cron 문서와 `cron/scheduler.py`를 보면 due job마다 fresh `AIAgent` session을 만들고, skill을 주입하고, prompt를 실행한 뒤, 결과를 origin chat, local file, Telegram, Discord, Slack, Email 등으로 보낸다. no-agent mode도 있어서 단순 script stdout delivery는 LLM 없이 처리할 수 있다.

GitHub 이슈를 보면 이 사용 방식이 단지 문서상의 상상이 아니라는 점도 보인다. Telegram daily briefing, Telegram rich message table 문제, Slack Block Kit 요구, Email gateway subject isolation, cron `deliver=origin` target loss, sandbox file transfer, Telegram photo caching, provider switch poisoning, Windows/WSL remote backend, macOS launchd service 같은 이슈가 있었다. 일부는 이미 closed 상태지만, Hermes가 개인 CLI 도구를 넘어 메시징과 자동화 표면에서 실제로 부딪히고 있다는 근거로는 충분하다.

## 좋은 점

가장 좋게 본 부분은 지속성이다. 요즘 agent 도구는 많지만, 대부분은 한 세션 안에서 꽤 똑똑하게 움직이는 데 집중한다. Hermes는 한 발 더 나아가 “다음 세션에서도 이 맥락을 어떻게 다시 꺼낼 것인가”를 중요하게 본다. Memory, session search, skills, cron이 모두 그 방향으로 붙어 있다.

운영 디테일도 생각보다 좋았다. 예를 들어 `model_tools.py`는 async tool handler를 sync 경로에서 돌리기 위해 persistent event loop와 per-worker loop를 쓴다. 그냥 매번 `asyncio.run()`을 호출하면 cached httpx 또는 AsyncOpenAI client가 닫힌 event loop에 묶여 `Event loop is closed` 류의 오류를 만들 수 있다. 이런 문제는 문서만 읽어서는 잘 안 보이고, 실제로 오래 돌려본 코드에서 자주 드러난다.

MCP discovery를 다루는 방식도 인상적이었다. `model_tools.py` 주석은 MCP discovery를 import side effect에서 제거한 이유를 설명한다. gateway가 asyncio event loop 안에서 처음 메시지를 받을 때 MCP discovery가 최대 120초까지 blocking되면 Discord나 Telegram heartbeat가 얼 수 있기 때문이다. 장기 실행 gateway에서는 이런 작은 blocking 하나가 전체 사용성을 망친다.

`tools/registry.py`의 availability hardening도 볼 만하다. Docker daemon, Modal SDK, Playwright binary 같은 외부 상태 probe를 매번 새로 하지 않고 TTL cache를 둔다. 더 중요한 것은 최근 성공 직후의 일시 실패를 flake로 보고 last-good true를 짧게 유지한다는 점이다. 순간적인 Docker probe timeout 때문에 subagent에서 file/terminal toolset이 갑자기 사라지는 일을 줄이는 장치다.

세션 저장소 쪽도 대충 만들지 않았다. `hermes_state.py`는 SQLite WAL mode를 기본으로 쓰되, NFS, SMB, 일부 FUSE처럼 WAL locking이 깨지는 환경에서는 DELETE journal mode로 fallback한다. macOS에서는 checkpoint full fsync barrier도 적용한다. “SQLite에 저장한다”는 말은 흔하지만, 파일 시스템별 locking 문제까지 고려한 구현은 덜 흔하다.

Cron 쪽에서는 fail-closed 성향이 마음에 들었다. 공식 cron 문서는 provider/model을 명시하지 않은 job이 생성 시점의 provider/model snapshot을 잡고, 나중에 global default가 바뀌면 fail closed한다고 설명한다. `cron/scheduler.py`도 cron-spawned agent에서 `cronjob`, `messaging`, `clarify` toolset을 항상 비활성화한다. unattended job이 cron을 무한 생성하거나, 사용자 응답을 기다리며 멈추는 일을 막기 위한 판단이다.

## 조심해야 할 점

Hermes의 가장 큰 리스크는 기능이 많다는 것 자체다. 이 도구는 terminal, file, browser, messaging, cron을 다룬다. 설정을 잘못하면 agent가 로컬 파일을 읽고 쓰고, 명령을 실행하고, 브라우저를 조작하고, 메시징 플랫폼에서 외부 요청을 받아 처리하고, 예약 작업으로 무인 실행할 수 있다. 설치 경험은 AI assistant처럼 보여도, 실제 배치는 권한 있는 worker를 띄우는 일에 가깝다.

Approval 설정은 특히 조심해야 한다. 보안 문서는 `approvals.mode`가 `manual`, `smart`, `off`를 지원한다고 설명한다. `off`는 모든 approval check를 끄는 모드이고, YOLO도 위험 명령 승인 prompt를 우회한다. hardline blocklist가 항상 남아 있기는 하지만, 그것은 최후의 바닥일 뿐이다. 처음부터 `off`나 YOLO로 쓰는 것은 추천하기 어렵다.

Container backend도 오해하기 쉽다. 보안 문서는 Docker, Singularity, Modal, Daytona backend에서는 container가 security boundary라 dangerous command check가 skip된다고 설명한다. 그런데 tools 문서를 같이 보면 Docker backend는 fresh container per command가 아니라 process lifetime 동안 유지되는 persistent sandbox다. 패키지 설치, cwd, 파일, 환경 조정이 다음 tool call과 subagent에 남는다. 이건 편리하지만, 오염도 같이 남는다는 뜻이다.

Memory와 skills는 장점이면서 동시에 drift 위험이다. Memory는 세션 시작 prompt에 들어가므로 틀린 가정이 저장되면 다음 세션을 계속 오염시킨다. Skills는 더 강하다. 잘못된 절차가 `SKILL.md`로 저장되면 agent가 같은 실수를 반복할 수 있다. 그래서 `memory.write_approval: true`, skill folder Git 관리, 민감 skill 보호, `/learn` 결과 리뷰가 중요하다.

메시징 표면은 UX 리스크도 크다. 실제 이슈를 보면 Telegram Markdown/table escaping, Telegram image caching timeout, Email notification volume, Slack Block Kit 부족, origin delivery target loss 같은 문제가 나온다. Agent 본체가 아무리 좋아도 플랫폼별 delivery reliability와 session isolation이 흔들리면 사용자는 “대충 불안정하다”고 느낀다.

마지막으로 toolset을 보안 경계로 과신하면 안 된다. Toolset은 모델에게 노출되는 schema를 줄이는 장치다. OS 권한, 네트워크 접근, 파일 시스템 권한을 완전히 격리하는 장치는 아니다. 실제 보안 경계는 Docker/SSH/Modal 같은 backend, gateway allowlist, approval mode, env passthrough 최소화, workdir 제한, API key scope 제한과 함께 만들어야 한다.

## 언제 쓰면 좋은가

Hermes가 잘 맞는 사람은 여러 프로젝트와 반복 업무를 장기간 다루는 사람이다. 매번 같은 환경 설명을 다시 해야 하고, 이전 대화에서 결정한 내용을 찾고 싶고, Telegram이나 Slack에서 작업을 던지고 싶고, 매일 또는 매시간 리서치와 운영 점검을 자동화하고 싶다면 Hermes의 구조가 잘 맞는다.

예를 들어 개인 개발자라면 아침에는 Telegram으로 시장/뉴스/프로젝트 브리핑을 받고, 낮에는 CLI에서 repo 테스트 실패를 분석하고, 저녁에는 Slack에서 긴 이슈 요약을 시키는 식으로 쓸 수 있다. 이때 중요한 것은 채널이 많다는 사실 자체가 아니라, 같은 runtime 위에서 memory와 session search가 계속 쌓인다는 점이다.

개발자에게는 agent harness 연구 대상으로도 가치가 있다. Provider resolution, tool registry, progressive skill disclosure, persistent session store, gateway authorization, cron fail-closed 같은 구현을 실제 코드로 볼 수 있다. AI agent runtime을 만들고 싶은 사람이라면 단순 기능 목록보다 내부 구현을 읽는 쪽이 더 도움이 된다.

반대로 단순한 질문 답변, deterministic한 SaaS integration, 엄격한 감사 환경, 공개 workspace에 바로 agent를 열어야 하는 팀에는 조심스럽다. Zapier나 n8n 같은 workflow tool은 예측 가능성이 강하고, Hermes는 LLM 판단과 권한 실행이 섞인다. 열린 문제를 다루는 데는 강하지만, 감사 가능한 deterministic workflow에는 별도 설계가 필요하다.

내가 도입한다면 이렇게 갈 것 같다.

| 단계 | 목표 | 권장 설정 |
| --- | --- | --- |
| 0 | 로컬 실험 | messaging gateway 끄기, `terminal.backend=docker` 또는 `ssh`, approval manual |
| 1 | 단발 CLI 작업 | `web,file,terminal,todo,skills`처럼 좁은 toolset |
| 2 | Memory 안정화 | `memory.write_approval=true`, `/memory pending` 리뷰 |
| 3 | Skills 축적 | 반복 workflow를 3번 이상 수행한 뒤 `/learn`, 결과 Git diff 리뷰 |
| 4 | Gateway 연결 | allowlist 필수, trusted DM부터, public/group은 read-only toolset |
| 5 | Cron 자동화 | read-only report부터, provider/model pinning, delivery fallback |

## 주니어 개발자가 배울 점

Hermes에서 배울 첫 번째 점은 agent 품질이 모델 성능만으로 결정되지 않는다는 것이다. 어떤 도구를 보여줄지, 어떤 상태를 기억할지, 실패했을 때 어떻게 fallback할지, 결과를 어디에 저장할지 같은 런타임 설계가 실제 사용감을 만든다.

두 번째는 “기억”에도 층이 있다는 점이다. Memory는 매번 prompt에 들어가는 작고 정확한 사실이어야 한다. Session search는 필요할 때 과거 대화를 다시 찾는 recall layer다. Skills는 반복 절차를 저장하는 executable documentation에 가깝다. 이 셋을 구분하면 다른 AI 도구를 볼 때도 구조가 훨씬 잘 보인다.

세 번째는 자동화와 권한을 따로 보면 안 된다는 점이다. Cron은 편하지만 unattended execution이다. Messaging gateway는 편하지만 외부 입력이 agent 권한으로 들어오는 길이다. Terminal backend는 편하지만 host 손상 가능성과 secret 노출 가능성을 함께 가져온다.

네 번째는 좋은 코드베이스가 운영 이슈의 흔적을 남긴다는 점이다. Hermes 코드에서 인상적인 부분은 화려한 기능 이름보다 event loop 재사용, WAL fallback, last-good availability cache, provider/model drift fail-closed, cron prompt injection scan 같은 작은 방어선이었다. 이런 장치는 보통 실제 실패를 겪어야 생긴다.

마지막으로 “작게 시작하는 설정”이 실력이다. 처음부터 full tools, local backend, YOLO, gateway allow-all, auto memory write를 켜는 것은 빠르지만 위험하다. 좁은 toolset, sandboxed backend, approval, allowlist, staged memory review로 시작해 필요한 권한만 넓히는 편이 훨씬 프로답다.

## 내 결론

Hermes Agent는 단순 코딩 에이전트보다 더 넓은 방향을 본다. CLI에서만 일하는 도구가 아니라 gateway와 cron을 통해 계속 살아 있고, memory와 skills를 통해 다음 작업에 영향을 주며, 여러 provider와 tool backend를 하나의 agent loop로 묶는다.

나는 이 프로젝트를 “개인 AI 운영체제”라고까지 부르는 표현은 조금 과하다고 본다. 하지만 “개인 AI 워커 런타임”이라는 표현은 꽤 잘 맞는다. `AIAgent`로 수렴하는 구조, registry 기반 tool system, SQLite/FTS5 session recall, cron의 fresh agent session 모델은 agent runtime을 공부하는 사람에게 충분히 읽을 가치가 있다.

다만 Hermes를 “믿고 맡기는 비서”처럼 소개하면 위험하다. 이 도구는 terminal/file/browser/messaging/cron 권한을 가진 worker다. 설치 직후부터 강하게 쓰기보다 Docker 또는 SSH backend, manual approval, memory write approval, gateway allowlist, 좁은 toolset으로 시작하는 편이 맞다. Cron은 처음엔 no-agent 또는 read-only report부터, skill은 Git으로 관리하면서 키우는 쪽이 안전하다.

한 줄로 정리하면 이렇다. Hermes는 믿고 맡기는 제품이라기보다, 권한을 설계하며 키우는 개인 AI 워커 런타임이다. 그 관점으로 접근하면 강력하고, 그 관점을 놓치면 위험하다.
