Context Refinement System의 출발점은 단순하다. 사용자가 말을 완벽하게 하지 않아도 AI가 목표, 배경, 제약, 출력 범위를 제대로 분리해 답하게 만들자는 것이다.

많은 AI 시스템은 사용자에게 좋은 프롬프트를 요구한다. 목표, 배경, 제약, 출력 형식, 제외 조건을 매번 정확히 써야 결과가 좋아진다. 하지만 실제 사용자는 그렇게 말하지 않는다. “이거 좀 이상한데 방향만 봐줘”처럼 짧고 거친 입력을 던진다.

이 시스템은 그 입력을 무조건 긴 프롬프트로 바꾸지 않는다. 핵심은 Hook은 항상 실행되는 가벼운 판단기로 두고, 실제 정리와 검증은 필요한 경우에만 Skill이 수행하게 하는 것이다.

## 문제: 컨텍스트 오염

사용자 입력에는 여러 층이 섞인다. 진짜 목표, 배경 설명, 감정 표현, 예시, 걱정거리, 후보 해결책, 제외 조건이 한 문장에 들어간다. AI가 그중 하나를 잘못 중심에 놓으면 답변은 원래 목적에서 벗어난다.

컨텍스트 오염은 작업 과정에서도 생긴다. 코드베이스 탐색, 웹 리서치, 로그 분석, GitHub 이슈 확인처럼 노이즈가 큰 작업을 메인 thread에 모두 쌓으면 원래 목표보다 중간 산출물에 끌리기 쉽다.

따라서 시스템은 두 가지를 해야 한다. 입력 의미를 분리하고, 노이즈가 큰 작업은 subagent나 별도 레이어로 격리해야 한다.

## 핵심 설계

전체 흐름은 다음과 같다.

1. User Raw Input
2. Lightweight Preflight Hook
3. 명확하면 그대로 진행
4. 애매하면 Prompt Normalizer Skill 호출
5. Intent Router
6. 필요한 Domain Skill 또는 Brainstorming 또는 Grill-plan 호출
7. Main AI Response
8. Answer Guard
9. Final Response

Lightweight Preflight Hook은 입력을 다시 쓰지 않는다. 이 입력이 명확한지, 정리가 필요한지, 위험한 작업인지, 어떤 라우팅이 필요한지만 가볍게 판단한다.

Prompt Normalizer Skill은 on-demand로만 호출된다. 사용자의 말을 목표, 배경, 제약, 제외 조건, 성공 기준, 열린 질문으로 분리한다. 중요한 점은 사용자가 던진 후보 해결책을 확정 요구사항으로 바꾸지 않는 것이다.

Intent Router는 작업을 writing, dev, design, resume, research, product 같은 도메인으로 보낸다. 도메인별 Skill은 작업 방법을 담당하고, Hook은 언제 개입할지를 담당한다.

Brainstorming은 기능 아이디어가 흐리거나 제품 방향을 잡아야 할 때만 쓴다. Grill-plan은 이미 계획이 있지만 중요한 결정을 검증해야 할 때 사용한다. Answer Guard는 마지막에 답변이 원래 요청에서 벗어나지 않았는지 확인한다.

## 하지 말아야 할 것

이 시스템은 모든 입력을 매번 긴 프롬프트로 재작성하지 않는다. 비용도 커지고, 사용자의 뉘앙스도 사라지고, 짧은 대화까지 딱딱해진다.

또한 모든 요청에 질문하지 않는다. 사용자가 원하는 것은 프롬프트 작성 교육이 아니라, 대충 말해도 의도를 잘 잡아주는 경험이다. 질문은 필요한 곳에서만 해야 한다.

모든 아이디어를 PRD로 만드는 것도 목표가 아니다. 가벼운 문장 정리, 간단한 코드 수정, 빠른 의견 요청까지 무거운 설계 프로세스로 보내면 사용 경험이 나빠진다.

## 구현 관점

구현은 단계적으로 하는 편이 좋다. MVP에서는 Preflight Hook, Prompt Normalizer, Answer Guard부터 시작한다. 그다음 Intent Router와 핵심 Domain Skills를 붙인다. 이후 Brainstorming, Grill-plan, eval, subagents, docs layer로 확장한다.

평가도 중요하다. messy input test set을 만들고, 라우팅 정확도, 목표 보존, 과잉 질문률, 답변 이탈률, 사용자 수정률을 측정해야 한다. 이런 시스템은 감으로만 좋아졌다고 말하기 어렵다.

보안 측면에서는 prompt injection, tool guard, 문서 오염 방지가 필요하다. 사용자의 입력을 정리하는 시스템 자체가 권한 있는 작업을 실행한다면, 무엇을 신뢰하고 무엇을 격리할지 명확해야 한다.

## 정리

Context Refinement System은 프롬프트를 예쁘게 고쳐 쓰는 장치가 아니다. 사용자의 거친 입력을 실행 가능한 작업 요청으로 바꾸고, 필요한 곳에만 개입하며, 최종 답변이 원래 목표를 벗어나지 않게 하는 작업 운영체제에 가깝다.

AI가 더 강해질수록 이런 중간 레이어의 가치는 커진다. 모델이 알아서 잘할 것이라는 기대보다, 모델이 잘할 수밖에 없는 입력과 라우팅과 검증 구조를 만드는 편이 더 안정적이다.
