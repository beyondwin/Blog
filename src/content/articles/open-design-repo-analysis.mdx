`nexu-io/open-design`은 자연어 디자인 브리프를 HTML, 프로토타입, 덱, 템플릿, 디자인 시스템, 미디어 산출물로 바꾸는 로컬 우선 디자인 생성 제품이다. 표면적으로는 Claude Design의 오픈소스 대안처럼 보이지만, 내부 구조를 보면 단순한 디자인 생성기보다 디자인 에이전트 런타임에 가깝다.

핵심은 Open Design이 직접 모든 모델 루프를 소유하지 않는다는 점이다. 사용자의 로컬 환경에 이미 설치된 Claude Code, Codex, Cursor Agent, Gemini CLI, OpenCode 같은 도구를 감지하고 실행한다. 즉 제품의 역할은 모델이 아니라 작업 셸이다.

## 제품 정체성

Open Design은 네 가지 축으로 이해할 수 있다.

첫째, local-first다. 데몬과 산출물은 사용자 머신에 있고, API 키와 인증 상태도 사용자의 환경에 남는다.

둘째, BYO-agent다. 자체 에이전트만 쓰게 하지 않고 사용자가 이미 인증해 둔 CLI를 활용한다.

셋째, skills as files다. 디자인 기능은 `SKILL.md` 기반 폴더로 배포된다. 스킬은 Markdown, frontmatter, assets, references를 포함할 수 있다.

넷째, design systems as files다. 브랜드와 제품 스타일은 `DESIGN.md`로 표현된다. 데몬은 이 파일을 읽어 프롬프트에 주입하고, UI에는 요약과 swatch를 보여준다.

이 구조 덕분에 Open Design은 특정 모델 업체나 특정 UI 생성 방식에 고정되지 않는다. 반대로 외부 CLI의 버전, 권한 모델, 스트리밍 포맷 변화에는 민감해진다.

## 전체 아키텍처

구조는 크게 Next.js 웹앱, 로컬 Express 데몬, 에이전트 CLI 레이어, 런타임 데이터 저장소로 나뉜다.

웹앱은 프로젝트 생성, 채팅, 에이전트 선택, 스킬과 디자인 시스템 선택, 파일 트리, preview, comment와 refinement UI를 담당한다.

로컬 데몬은 REST와 SSE API를 제공하고, 에이전트 CLI를 감지하고 실행하며, stdout과 stderr 스트림을 해석한다. 프로젝트 파일, 대화, 메시지, preview comment, deployment 메타데이터도 관리한다.

에이전트 레이어는 Claude Code, Codex, Gemini CLI, Cursor Agent, OpenCode, Copilot 등 여러 CLI를 서로 다른 실행 방식과 스트림 포맷으로 어댑터화한다.

저장소는 `.od` 런타임 데이터 디렉터리와 SQLite 메타데이터를 중심으로 한다. 생성된 산출물은 파일로 남고, 메타데이터는 DB에 저장된다.

## 강점

가장 큰 강점은 명확한 차별화다. Open Design은 “또 하나의 AI 디자인 앱”이 아니라, 사용자가 이미 쓰는 에이전트를 연결하는 로컬 작업 환경이다.

파일 기반 확장 모델도 강하다. `SKILL.md`와 `DESIGN.md`는 사람이 읽고 수정하기 쉽다. 제품 내부 설정 화면에 갇히지 않고, 버전 관리와 공유가 가능하다.

프롬프트 품질에 대한 문제의식도 분명하다. 디자인 산출물에서 흔히 나타나는 AI slop을 줄이기 위한 craft rule, preflight directive, discovery form 같은 구조가 있다.

local-first와 BYOK는 보안과 신뢰 측면에서 장점이다. 디자인 작업물과 인증 정보가 외부 서비스에 과하게 종속되지 않는다.

## 리스크

첫 번째 리스크는 데몬의 비대화다. 로컬 데몬이 에이전트 실행, DB, 라우팅, 파일, preview, 배포, media generation까지 많은 책임을 갖게 되면 유지보수 경계가 흐려질 수 있다.

두 번째 리스크는 외부 CLI 의존성이다. 각 에이전트의 옵션, 출력 포맷, 권한 모델이 바뀌면 Open Design도 영향을 받는다. 어댑터별 회귀 테스트가 중요하다.

세 번째 리스크는 prompt stack 회귀 테스트의 어려움이다. 스킬, 디자인 시스템, craft rule, user prompt가 결합되면 결과 품질이 작은 변경에도 흔들릴 수 있다.

네 번째 리스크는 문서 drift다. 초기 프로젝트는 README, 소스, 실제 동작, 릴리스 노트가 빠르게 어긋날 수 있다. 분석 문서에서도 스킬과 디자인 시스템 개수 표기가 일관되지 않은 점이 보였다.

마지막으로 권한 posture가 중요하다. 로컬 CLI를 실행하고 파일을 만들며 preview iframe을 띄우는 제품은 편리하지만, 샌드박스와 read-only 경계를 명확히 해야 한다.

## 도입 방식

개인은 실험용 로컬 디자인 셸로 써볼 수 있다. 기존에 Claude Code나 Codex를 이미 사용하고 있다면, 디자인 산출물을 더 체계적으로 만들고 비교하는 데 도움이 된다.

팀은 바로 전면 도입하기보다 제한된 디자인 시스템과 몇 개의 스킬로 시작하는 편이 좋다. 에이전트 어댑터, 권한, 산출물 저장 위치, 리뷰 프로세스, 배포 경계를 먼저 정해야 한다.

## 정리

Open Design의 본질은 디자인 생성 앱보다 디자인 에이전트 런타임이다. 자연어 브리프, 스킬, 디자인 시스템, 로컬 에이전트 CLI, preview, 파일 저장을 하나의 작업 흐름으로 묶는다.

이 접근은 AI 제품의 방향을 잘 보여준다. 경쟁력은 모델 호출 버튼이 아니라, 어떤 작업 맥락을 만들고 어떤 산출물을 남기며 어떤 경계 안에서 에이전트를 실행하느냐에 있다.
